# FAQ

### 얼마나 많은 코드를 작성해야 하나요?

다음은 `git diff --stat`로 생성한 레퍼런스 해법의 요약입니다.
마지막 줄은 전체 추가/삭제 줄 수를 보여 줍니다. 수정된 한 줄은 추가와 삭제가 각각 하나씩으로 계산됩니다.
레퍼런스 해법은 가능한 해법 중 하나일 뿐입니다. 다른 해법도 많이 가능하며, 그중 상당수는 레퍼런스 해법과 크게 다를 수 있습니다. 훌륭한 해법 중에는 레퍼런스 해법이 수정한 모든 파일을 건드리지 않는 것도 있고, 반대로 레퍼런스 해법이 건드리지 않은 파일을 수정하는 것도 있을 수 있습니다. 또한 여기에 extra 요구사항 구현도 포함되어 있습니다. 참고로 약 150줄은 extra와 관련되어 있습니다.

```
src/include/threads/thread.h   |  23 ++
src/include/userprog/syscall.h |   3 +
src/threads/thread.c           |   5 +
src/userprog/exception.c       |   4 +
src/userprog/process.c         | 355 +++++++++++++++++++++++++++++++++++++++++-
src/userprog/syscall.c         | 429 ++++++++++++++++++++++++++++++++++++++++++-
6 files changed, 782 insertions(+), 37 deletions(-)
```

### `pintos -p file -- -q`를 실행하면 커널이 항상 panic을 일으킵니다.

파일 시스템을 포맷했나요(`pintos -f`)?

파일 이름이 너무 길지는 않나요? 파일 시스템은 파일 이름을 14자로 제한합니다.

`pintos -p ../../examples/echo -- -q` 같은 명령은 이 제한을 초과합니다.

대신 `pintos -p ../../examples/echo:echo -- -q`를 사용해 `echo`라는 이름으로 파일을 넣으세요.

파일 시스템이 가득 찼나요?

파일 시스템에 이미 16개 파일이 들어 있지는 않나요? 기본 Pintos 파일 시스템은 16개 파일 제한이 있습니다.

파일 시스템이 너무 단편화되어 있어서 여러분의 파일이 들어갈 연속 공간이 부족할 수도 있습니다.

### `pintos -p ../file --`를 실행했는데 'file'이 복사되지 않습니다.

기본적으로 파일은 명령에서 참조한 이름 그대로 기록됩니다. 따라서 이 경우 복사된 파일의 이름은 `'../file'`이 됩니다. 아마 `pintos -p ../file:file --`를 대신 실행하고 싶을 것입니다.

### 모든 사용자 프로그램이 page fault와 함께 죽습니다.

인자 전달을 아직 구현하지 않았거나(혹은 제대로 구현하지 않았을 때) 이런 일이 발생합니다. 사용자 프로그램용 기본 C 라이브러리는 스택에서 argv를 읽으려 합니다. 스택이나 레지스터가 제대로 설정되어 있지 않으면 이 과정에서 page fault가 발생합니다.

### 모든 사용자 프로그램이 `system call!`과 함께 죽습니다.

그다음 단계를 보기 전에 시스템 콜을 먼저 구현해야 합니다. 정상적인 프로그램이라면 최소한 하나의 시스템 콜(`exit()`)은 호출하고, 대부분은 그보다 더 많은 시스템 콜을 호출합니다. 특히 `printf()`는 write 시스템 콜을 호출합니다. 기본 시스템 콜 핸들러는 단지 `system call!`을 출력하고 프로그램을 종료합니다. 그전까지는 `hex_dump()`를 사용해 인자 전달이 제대로 구현되었는지 확인할 수 있습니다.

### 사용자 프로그램을 어떻게 디스어셈블하나요?

objdump 유틸리티는 사용자 프로그램 전체나 object file을 디스어셈블할 수 있습니다. `objdump -d file` 형태로 실행하세요. 개별 함수를 디스어셈블하려면 GDB의 disassemble 명령을 사용할 수 있습니다.

### 왜 많은 C include 파일이 Pintos 프로그램에서는 동작하지 않나요?

### Pintos 프로그램에서 libfoo를 사용할 수 있나요?

우리가 제공하는 C 라이브러리는 매우 제한적입니다. 실제 운영체제의 C 라이브러리라면 당연히 제공할 것으로 기대되는 많은 기능이 들어 있지 않습니다. C 라이브러리는 해당 운영체제(그리고 아키텍처)에 맞게 특별히 빌드되어야 합니다. I/O와 메모리 할당을 위해 시스템 콜을 호출해야 하기 때문입니다. (물론 모든 함수가 그런 것은 아니지만, 보통 라이브러리는 하나의 단위로 컴파일됩니다.)

여러분이 원하는 라이브러리는 Pintos가 구현하지 않은 C 라이브러리 기능에 의존할 가능성이 큽니다. 따라서 Pintos에서 동작하게 만들려면 어느 정도의 포팅 작업이 필요할 것입니다. 특히 Pintos 사용자 프로그램용 C 라이브러리에는 `malloc()` 구현이 없습니다.

`src/examples/Makefile`을 수정한 뒤 `make`를 실행하세요.

### tid_t와 pid_t의 차이는 무엇인가요?

tid_t는 커널 스레드를 식별합니다. 그 스레드는 사용자 프로세스를 실행 중일 수도 있고(`process_fork()`로 생성된 경우), 아닐 수도 있습니다(`thread_create()`로 생성된 경우). 이것은 커널에서만 사용하는 데이터 타입입니다.
pid_t는 사용자 프로세스를 식별합니다. 사용자 프로세스와 커널 모두 exec 및 wait 시스템 콜에서 이 값을 사용합니다. tid_t와 pid_t에는 여러분이 적절하다고 생각하는 타입을 자유롭게 선택할 수 있습니다. 기본값은 둘 다 int입니다. 둘을 일대일 대응으로 만들어 같은 값이 같은 프로세스를 가리키게 해도 되고, 더 복잡한 매핑을 사용해도 됩니다. 선택은 여러분에게 달려 있습니다.

### `struct file *`를 그냥 캐스팅해서 파일 디스크립터로 써도 되나요?

### `struct thread *`를 그냥 캐스팅해서 `pid_t`로 써도 되나요?

아니요. `pid_t`와 파일 디스크립터는 포인터 타입보다 작으므로, 그런 식으로 직접 캐스팅하는 방법은 적절하지 않습니다.

### 프로세스마다 열 수 있는 파일 수의 최대값을 정해도 되나요?

임의의 제한을 두지 않는 편이 더 좋습니다. 꼭 필요하다면 프로세스당 열린 파일 수를 128개로 제한할 수는 있습니다. 하지만 extra 요구사항까지 구현하고 싶다면 제한이 없어야 합니다.

### 열린 파일이 제거되면 어떻게 되나요?

표준 Unix 의미론으로 파일을 구현해야 합니다. 즉, 파일이 제거되더라도 그 파일을 가리키는 파일 디스크립터를 가진 프로세스는 계속 그 디스크립터를 사용할 수 있어야 합니다. 이는 그들이 그 파일에서 읽고 쓸 수 있다는 뜻입니다. 파일은 이름을 잃게 되고 다른 프로세스는 더 이상 그 파일을 열 수 없지만, 그 파일을 가리키는 모든 파일 디스크립터가 닫히거나 머신이 종료될 때까지는 계속 존재합니다.

### 4 kB보다 큰 스택 공간이 필요한 사용자 프로그램은 어떻게 실행하나요?

각 프로세스에 대해 한 페이지보다 큰 스택 공간을 할당하도록 스택 설정 코드를 수정해도 됩니다. 다음 프로젝트에서는 더 나은 해법을 구현하게 됩니다.
