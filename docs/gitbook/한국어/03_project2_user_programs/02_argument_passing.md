#### x86-64 호출 규약 !!!!!!!!!

이 절에서는 64비트 x86-64 Unix 구현에서 일반 함수 호출에 사용하는 호출 규약의 중요한 부분을 요약합니다. 간결함을 위해 일부 세부 사항은 생략합니다. 더 자세한 내용은 [System V AMD64 ABI](https://en.wikipedia.org/wiki/X86_calling_conventions#System_V_AMD64_ABI)를 참고하세요.

호출 규약은 다음과 같이 동작합니다.

1. 사용자 수준 애플리케이션은 정수 인자를 전달할 때 `%rdi`, `%rsi`, `%rdx`, `%rcx`, `%r8`, `%r9` 순서의 레지스터를 사용합니다.
2. 호출자는 다음 명령의 주소(반환 주소)를 스택에 push한 뒤 피호출자의 첫 번째 명령으로 jump합니다. x86-64 명령 하나인 `CALL`이 이 두 작업을 모두 수행합니다.
3. 피호출자가 실행됩니다.
4. 피호출자에게 반환값이 있으면 레지스터 RAX에 저장합니다.
5. 피호출자는 반환 주소를 스택에서 pop하고, x86-64의 `RET` 명령으로 그 위치로 jump하여 반환합니다.

세 개의 int 인자를 받는 함수 `f()`를 생각해 봅시다. 아래 그림은 `f(1, 2, 3)`으로 호출되었다고 가정할 때, 위의 step 3이 시작되는 시점에 피호출자 기준으로 보이는 스택 프레임과 레지스터 상태를 보여 줍니다. 초기 스택 주소는 임의입니다.

```
                             +----------------+
stack pointer --> 0x4747fe70 | return address |
                             +----------------+
RDI: 0x0000000000000001 | RSI: 0x0000000000000002 | RDX: 0x0000000000000003
```

#### 프로그램 시작 세부 사항

사용자 프로그램용 Pintos C 라이브러리는 `lib/user/entry.c`에 있는 `_start()`를 사용자 프로그램의 진입점으로 지정합니다. 이 함수는 `main()`을 감싸는 wrapper이며, `main()`이 반환하면 `exit()`를 호출합니다.

```c
void
_start (int argc, char *argv[]) {
    exit (main (argc, argv));
}
```

커널은 사용자 프로그램 실행을 시작시키기 전에 초기 함수의 인자를 레지스터에 배치해야 합니다. 인자는 일반 호출 규약과 같은 방식으로 전달됩니다.

다음 예시 명령의 인자를 어떻게 처리할지 생각해 봅시다.
`/bin/ls -l foo bar`.

1. 명령을 `/bin/ls`, `-l`, `foo`, `bar`라는 단어들로 나눕니다.
2. 단어들을 스택의 맨 위에 배치합니다. 나중에 포인터를 통해 참조되므로 순서는 중요하지 않습니다.
3. 각 문자열의 주소와 null 포인터 sentinel을 스택에 오른쪽에서 왼쪽 순서로 push합니다. 이것들이 argv의 원소들입니다. null 포인터 sentinel은 C 표준이 요구하는 대로 `argv[argc]`가 null 포인터가 되도록 보장합니다. 이 순서를 따르면 `argv[0]`가 가장 낮은 가상 주소에 놓입니다. word-aligned 접근은 unaligned 접근보다 빠르므로, 성능을 위해 첫 번째 push 전에 스택 포인터를 8의 배수로 내리는 것이 좋습니다.
4. `%rsi`가 `argv`(`argv[0]`의 주소)를 가리키게 하고, `%rdi`에는 `argc`를 넣습니다.
5. 마지막으로 가짜 "return address"를 push합니다. 진입 함수는 실제로 반환하지 않더라도, 스택 프레임 구조는 다른 함수와 같아야 합니다.

아래 표는 사용자 프로그램이 시작되기 직전의 스택 상태와 관련 레지스터를 보여 줍니다.
스택은 아래 방향으로 자랍니다.

|주소|이름|데이터|타입|
| ----------| ------------| -----------| -----------|
|0x4747fffc|argv[3][...]|'bar\0'|char[4]|
|0x4747fff8|argv[2][...]|'foo\0'|char[4]|
|0x4747fff5|argv[1][...]|'-l\0'|char[3]|
|0x4747ffed|argv[0][...]|'/bin/ls\0'|char[8]|
|0x4747ffe8|워드 정렬|0|uint8_t[]|
|0x4747ffe0|argv[4]|0|char *|
|0x4747ffd8|argv[3]|0x4747fffc|char *|
|0x4747ffd0|argv[2]|0x4747fff8|char *|
|0x4747ffc8|argv[1]|0x4747fff5|char *|
|0x4747ffc0|argv[0]|0x4747ffed|char *|
|0x4747ffb8|반환 주소|0|void (*) ()|

> RDI: 4 | RSI: 0x4747ffc0

이 예시에서 스택 포인터는 `0x4747ffb8`로 초기화됩니다.
위에서 보았듯이, 여러분의 코드는 `include/threads/vaddr.h`에 정의된 `USER_STACK`에서 스택을 시작해야 합니다.

비표준 함수인 `hex_dump()`는 `<stdio.h>`에 선언되어 있으며, 인자 전달 코드를 디버깅할 때 유용할 수 있습니다.

#### 인자 전달을 구현하세요.

현재 `process_exec()`는 새 프로세스에 인자를 전달하는 기능을 지원하지 않습니다. 이 기능을 구현하려면 `process_exec()`를 확장해서, 단순히 프로그램 파일 이름만 받는 대신 공백을 기준으로 단어를 나누도록 하세요. 첫 번째 단어는 프로그램 이름이고, 두 번째 단어는 첫 번째 인자이며, 그다음도 같은 방식입니다. 즉, `process_exec("grep foo bar")`는 grep을 실행하면서 foo와 bar라는 두 인자를 전달해야 합니다.

명령줄 안에서 여러 공백은 하나의 공백과 같습니다. 따라서 `process_exec("grep foo bar")`는 원래 예제와 동일합니다. 명령줄 인자의 길이에는 합리적인 제한을 둘 수 있습니다. 예를 들어, 한 페이지(4 kB)에 들어갈 수 있는 인자만 허용하도록 제한할 수 있습니다. (이와 별개로 pintos 유틸리티가 커널에 전달할 수 있는 명령줄 인자에는 128바이트 제한이 있습니다.)

인자 문자열을 파싱하는 방법은 자유롭게 선택해도 됩니다. 막막하다면 `include/lib/string.h`에 선언되어 있고 `lib/string.c`에 자세한 주석과 함께 구현되어 있는 `strtok_r()`를 살펴보세요. 자세한 내용은 매뉴얼 페이지(`man strtok_r`)에서도 확인할 수 있습니다.
