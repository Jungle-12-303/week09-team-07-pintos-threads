# 01_introduction

# Project 2: 사용자 프로그램

이제 Pintos를 다뤄 보았고 그 인프라와 thread 패키지에 익숙해졌으니, 사용자 프로그램을 실행할 수 있게 하는 시스템 부분을 작업할 차례입니다. 기본 코드는 이미 사용자 프로그램을 로드하고 실행하는 기능을 지원하지만, 아직 I/O나 상호작용은 할 수 없습니다. 이번 프로젝트에서는 프로그램이 시스템 콜을 통해 OS와 상호작용할 수 있게 만들어야 합니다. 이번 과제에서는 `userprog` 디렉터리를 중심으로 작업하지만, Pintos의 거의 모든 다른 부분과도 상호작용하게 됩니다. 아래에서 관련 부분을 설명하겠습니다.

**반드시 project 1 제출물을 기반으로 project 2를 구현해야 합니다.**  project 1의 코드가 project 2 코드에 직접 영향을 주지는 않지만, 이 프로젝트는 누적형 프로젝트이므로 project 1의 테스트 케이스도 계속 통과해야 합니다.

또한 project 2에는 추가 도전 과제가 있습니다. **이는 선택 구현 사항일 뿐입니다.**  extra 과제에 대해서는 테스트 케이스 외에 별도의 skeleton code를 제공하지 않으므로, 설계는 전적으로 여러분에게 달려 있습니다. **추가 요구사항을 제출하고 테스트하려면 **​**​`userprog/Make.vars`​**​**를 수정해야 합니다.**

마지막으로, `TODO`가 없는 코드라고 해서 항상 수정할 필요가 없는 것은 아닙니다. 테스트 코드를 제외하면 project2의 어떤 소스 코드든 자유롭게 수정할 수 있습니다.

## 배경

지금까지 Pintos 아래에서 실행한 모든 코드는 운영체제 커널의 일부였습니다. 예를 들어, 지난 과제의 모든 테스트 코드는 시스템의 특권 영역에 완전히 접근할 수 있는 상태에서 커널의 일부로 실행되었습니다. 운영체제 위에서 사용자 프로그램을 실행하기 시작하면 더는 그렇지 않습니다. 이번 프로젝트는 그에 따른 결과를 다룹니다.

우리는 한 번에 하나 이상의 프로세스가 실행되도록 허용합니다. 각 프로세스는 하나의 스레드를 가집니다(멀티스레드 프로세스는 지원하지 않습니다). 사용자 프로그램은 자신이 기계 전체를 혼자 쓰고 있다고 가정한 채 작성됩니다. 즉, 여러 프로세스를 동시에 로드하고 실행할 때는 이 착각을 유지할 수 있도록 메모리, 스케줄링, 그리고 그 밖의 상태를 올바르게 관리해야 합니다.

이전 프로젝트에서는 테스트 코드를 커널에 직접 컴파일했기 때문에, 커널 내부에 특정 함수 인터페이스를 요구해야 했습니다. 이제부터는 사용자 프로그램을 실행해서 운영체제를 테스트하므로 훨씬 더 큰 자유도가 주어집니다. 여기 설명된 명세에 맞는 사용자 프로그램 인터페이스만 보장하면 되고, 그 제약만 지키면 커널 코드는 원하는 대로 재구성하거나 다시 작성해도 됩니다. 여러분의 코드는 절대로 `#ifdef VM`으로 둘러싸인 블록 안에 위치해서는 안 됩니다. 이 블록은 project 3에서 virtual memory subsystem을 활성화한 뒤 포함됩니다. 또한 `#ifndef VM`으로 둘러싸인 코드는 project 3에서 제외됩니다.

**시작하기 전에 **​**[동기화](../06_appendix/02_synchronization.md)**​**와 **​**[가상 주소](../06_appendix/04_virtual_address.md)**​**를 읽어 두기를 강력히 권장합니다.**

### 소스 파일 // 이런게 있구나 정도

어떤 프로그래밍을 하게 될지 전체 그림을 파악하는 가장 쉬운 방법은, 작업하게 될 각 부분을 차례대로 훑어보는 것입니다. `userprog` 안에는 파일 수가 많지 않지만, 실제 작업의 대부분은 다음 파일들에서 이루어집니다.

- `process.c`, `process.h`

  ELF 바이너리를 로드하고 프로세스를 시작합니다.
- `syscall.c`, `syscall.h`

  사용자 프로세스가 커널 기능을 사용하려고 할 때마다 시스템 콜을 호출합니다. 이 파일은 시스템 콜 핸들러의 skeleton입니다. 현재는 메시지를 출력하고 사용자 프로세스를 종료하기만 합니다. 이번 프로젝트의 part 2에서는 시스템 콜이 필요로 하는 나머지 모든 동작을 처리하는 코드를 여기에 추가하게 됩니다.
- `syscall-entry.S`

  시스템 콜 핸들러를 부트스트랩하는 작은 어셈블리 코드입니다. 이 코드를 이해할 필요는 없습니다.
- `exception.c`, `exception.h`

  사용자 프로세스가 특권이 필요한 연산이나 금지된 연산을 수행하면 `exception` 또는 `fault`로 커널에 트랩됩니다. 이 파일들은 그런 예외를 처리합니다. 현재는 모든 예외가 메시지를 출력한 뒤 프로세스를 종료하기만 합니다. project 2의 해법 중 일부는 이 파일의 `page_fault()`를 수정해야 하지만, 모두 그런 것은 아닙니다.
- `gdt.c`, `gdt.h`

  x86-64는 세그먼트 구조를 가진 아키텍처입니다. Global Descriptor Table (GDT)은 사용 중인 세그먼트를 설명하는 테이블입니다. 이 파일들은 GDT를 설정합니다. 어떤 프로젝트에서도 이 파일들을 수정할 필요는 없을 것입니다. GDT가 어떻게 동작하는지 궁금하다면 코드를 읽어 봐도 좋습니다.
- `tss.c`, `tss.h`

  Task-State Segment (TSS)는 x86 아키텍처의 task switching에 사용되었습니다. 하지만 x86-64에서는 task switching이 deprecated되었습니다. 그럼에도 ring switching 중 스택 포인터를 찾기 위해 TSS는 여전히 존재합니다.

  즉, 사용자 프로세스가 인터럽트 핸들러에 들어가면 하드웨어는 커널의 스택 포인터를 찾기 위해 tss를 조회합니다. 어떤 프로젝트에서도 이 파일들을 수정할 필요는 없을 것입니다. TSS가 어떻게 동작하는지 궁금하다면 코드를 읽어 봐도 좋습니다.

### 파일 시스템 사용하기 XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX

이번 프로젝트에서는 파일 시스템에서 사용자 프로그램을 로드하고, 구현해야 할 많은 시스템 콜이 파일 시스템을 다루기 때문에 파일 시스템 코드와도 인터페이스해야 합니다. 하지만 이번 프로젝트의 초점은 파일 시스템이 아니므로, `filesys` 디렉터리에 간단하지만 완전한 파일 시스템을 제공해 두었습니다. 파일 시스템을 사용하는 방법, 특히 그 많은 제한 사항을 이해하려면 `filesys.h`와 `file.h` 인터페이스를 살펴보는 것이 좋습니다.

**이번 프로젝트에서는 파일 시스템 코드를 수정할 필요가 없으므로**, 수정하지 않기를 권장합니다. 파일 시스템 작업은 이번 프로젝트의 핵심에서 벗어나게 만들 가능성이 큽니다.

**지금 파일 시스템 루틴을 올바르게 사용하는 습관을 들여 두면 project 4에서 파일 시스템 구현을 개선할 때 훨씬 수월해집니다.**  그때까지는 다음 제한 사항을 감수해야 합니다.

- 내부 동기화가 없습니다. 동시에 접근하면 서로 간섭합니다. 한 번에 하나의 프로세스만 파일 시스템 코드를 실행하도록 동기화를 사용해야 합니다.
- 파일 크기는 생성 시점에 고정됩니다. 루트 디렉터리는 파일로 표현되므로, 생성할 수 있는 파일 수 역시 제한됩니다.
- 파일 데이터는 하나의 extent로 할당됩니다. 즉, 하나의 파일 데이터는 디스크에서 연속한 섹터 범위를 차지해야 합니다. 따라서 파일 시스템을 오래 사용할수록 외부 단편화가 심각한 문제가 될 수 있습니다.
- 하위 디렉터리가 없습니다.
- 파일 이름은 14자로 제한됩니다.
- 작업 도중 시스템이 크래시하면 디스크가 자동으로 복구할 수 없는 방식으로 손상될 수 있습니다. 게다가 파일 시스템 복구 도구도 없습니다.

**중요한 기능 하나는 포함되어 있습니다.**

`filesys_remove()`에는 Unix와 유사한 의미론이 구현되어 있습니다. 즉, 파일이 제거될 때 그 파일이 열려 있으면 블록은 해제되지 않으며, 마지막으로 닫힐 때까지 그 파일을 열고 있는 모든 스레드가 계속 접근할 수 있습니다. 자세한 내용은 [열려 있는 파일 제거](08_faq.md)를 참고하세요.

project 1에서는 모든 테스트 프로그램이 이미 커널 이미지 안에 들어 있었지만, 이제는 테스트 프로그램(사용자 공간에서 실행됨)을 Pintos 가상 머신 안으로 넣어야 합니다. 편의를 위해 테스트 스크립트(`make check`)가 대부분의 경우 이 작업을 자동으로 처리하므로, 대개는 내부 동작을 이해할 필요가 없습니다. 다만 이를 알고 있으면 개별 테스트 케이스를 실행할 때 큰 도움이 됩니다.

파일을 Pintos 가상 머신 안에 넣으려면 먼저 파일 시스템 파티션이 들어 있는 시뮬레이션 디스크를 만들 수 있어야 합니다. 이를 위해 `pintos-mkdisk` 프로그램을 사용합니다. `userprog/build` 디렉터리에서 `pintos-mkdisk filesys.dsk 2`를 실행하세요. 이 명령은 `filesys.dsk`라는 이름의 시뮬레이션 디스크를 만들고, 그 안에 2 MB 크기의 Pintos 파일 시스템 파티션을 포함시킵니다. 그다음 `pintos` 뒤, `--` 앞에 `--fs-disk filesys.dsk`를 전달해서 디스크를 지정합니다(즉, `pintos --fs-disk filesys.disk -- KERNEL_COMMANDS...`). 여기서 `--`가 필요한 이유는 `--fs-disk`가 시뮬레이션 커널이 아니라 pintos 스크립트를 위한 옵션이기 때문입니다. 이후 커널 명령줄에 `-f -q`를 전달해 파일 시스템 파티션을 포맷하세요: `pintos SCRIPT_COMMANDS -- -f -q`. `-f` 옵션은 파일 시스템을 포맷하게 만들고, `-q`는 포맷이 끝나는 즉시 Pintos가 종료되게 만듭니다.

시뮬레이션 파일 시스템 안팎으로 파일을 복사하는 방법도 필요합니다. pintos의 `-p`("put") 및 `-g`("get") 옵션이 이를 담당합니다. `'file'`을 Pintos 파일 시스템 안으로 복사하려면 `pintos -p file -- -q`를 사용하세요. 원래 이름 대신 `newname`이라는 이름으로 넣고 싶다면 원본 파일 이름 뒤에 `:newname`을 붙이세요: `pintos -p file:newname -- -q`. 가상 머신 밖으로 파일을 복사하는 명령도 거의 같지만, `-p` 대신 `-g`를 사용합니다.

참고로, 이 명령들은 커널 명령줄에 특별한 `extract`와 `append` 명령을 전달하고, 특수한 시뮬레이션 "scratch" 파티션으로부터 복사하거나 그쪽으로 복사하는 방식으로 동작합니다. 구현 세부 사항이 궁금하다면 pintos 스크립트와 `filesys/fsutil.c`를 살펴보면 됩니다.

다음은 파일 시스템 파티션이 있는 디스크를 만들고, 파일 시스템을 포맷하고, 이번 프로젝트의 두 번째 테스트 케이스인 *args-single* 프로그램을 새 디스크에 복사한 뒤, 인자 `'onearg'`를 넘겨 실행하는 전체 흐름 요약입니다. (인자 전달을 구현하기 전까지는 동작하지 않습니다.) 테스트 케이스가 이미 빌드되어 있고 현재 디렉터리가 `userprog/build`라고 가정합니다.

```sh
pintos-mkdisk filesys.dsk 10
pintos --fs-disk filesys.dsk -p tests/userprog/args-single:args-single -- -q -f run 'args-single onearg'
```

나중에 다시 쓰거나 확인할 목적으로 파일 시스템 디스크를 남겨 두고 싶지 않다면, 이 네 단계를 하나의 명령으로 합칠 수도 있습니다. `--filesys-size=n` 옵션은 pintos를 실행하는 동안에만 유지되는, 대략 n 메가바이트 크기의 임시 파일 시스템 파티션을 만듭니다. Pintos 자동 테스트 스위트는 이 문법을 광범위하게 사용합니다.

```sh
pintos --fs-disk=10 -p tests/userprog/args-single:args-single -- -q -f run 'args-single onearg'
```

### 사용자 프로그램은 어떻게 동작하는가 !!!!!!!!!!!!!!!!!!  제일 중요함

Pintos는 메모리에 들어갈 수 있고 여러분이 구현한 시스템 콜만 사용하는 한, 일반적인 C 프로그램을 실행할 수 있습니다. 특히 이번 프로젝트에서 요구하는 시스템 콜들만으로는 메모리 할당을 할 수 없으므로 `malloc()`은 구현할 수 없습니다. 또한 Pintos는 부동소수점 연산을 사용하는 프로그램도 실행할 수 없습니다. 커널이 스레드 전환 시 프로세서의 부동소수점 장치를 저장하고 복원하지 않기 때문입니다.

Pintos는 `userprog/process.c`에 제공된 로더로 ELF 실행 파일을 로드할 수 있습니다. ELF는 Linux, Solaris, 그리고 많은 다른 운영체제에서 object file, shared library, executable에 사용하는 파일 형식입니다.

실제로는 x86-64 ELF 실행 파일을 출력할 수 있는 어떤 컴파일러와 링커라도 Pintos용 프로그램을 만드는 데 사용할 수 있습니다. (우리가 제공한 컴파일러와 링커만으로도 충분합니다.) 테스트 프로그램을 시뮬레이션 파일 시스템에 복사하기 전까지는 Pintos가 유용한 작업을 할 수 없다는 점을 바로 알 수 있을 것입니다. 파일 시스템에 여러 프로그램을 복사해 두기 전까지는 흥미로운 일을 해 볼 수 없습니다. 디버깅 중 `filesys.dsk`가 쓸모없을 정도로 망가지는 일이 가끔 생길 수 있으므로, 깨끗한 기준 파일 시스템 디스크를 하나 만들어 두고 필요할 때마다 복사해 쓰는 것도 좋습니다.

### 가상 메모리 레이아웃

Pintos의 가상 메모리는 두 영역으로 나뉩니다. 사용자 가상 메모리와 커널 가상 메모리입니다. 사용자 가상 메모리는 가상 주소 `0`에서 `KERN_BASE`까지이며, `KERN_BASE`는 `include/threads/vaddr.h`에 정의되어 있고 기본값은 `0x8004000000`입니다. 커널 가상 메모리는 그 나머지 가상 주소 공간을 차지합니다.

사용자 가상 메모리는 프로세스별로 독립적입니다. 커널이 한 프로세스에서 다른 프로세스로 전환할 때는 프로세서의 페이지 디렉터리 베이스 레지스터를 바꿔 사용자 가상 주소 공간도 함께 전환합니다(`thread/mmu.c`의 `pml4_activate()` 참고). `struct thread`에는 프로세스의 페이지 테이블을 가리키는 포인터가 들어 있습니다.

커널 가상 메모리는 전역입니다. 어떤 사용자 프로세스나 커널 스레드가 실행 중이든 항상 같은 방식으로 매핑됩니다. Pintos에서는 커널 가상 메모리가 `KERN_BASE`부터 물리 메모리에 일대일로 매핑됩니다. 즉, 가상 주소 `KERN_BASE`는 물리 주소 `0`에 접근하고, 가상 주소 `KERN_BASE + 0x1234`는 물리 주소 `0x1234`에 접근하는 식으로 머신의 물리 메모리 크기까지 이어집니다.

사용자 프로그램은 자신의 사용자 가상 메모리에만 접근할 수 있습니다. 커널 가상 메모리에 접근하려고 하면 `userprog/exception.c`의 `page_fault()`가 처리하는 페이지 폴트가 발생하고, 해당 프로세스는 종료됩니다. 커널 스레드는 커널 가상 메모리뿐 아니라 사용자 프로세스가 실행 중일 때 그 프로세스의 사용자 가상 메모리에도 접근할 수 있습니다. 하지만 커널 안에서도 매핑되지 않은 사용자 가상 주소에 접근하면 페이지 폴트가 발생합니다.

### 전형적인 메모리 레이아웃

개념적으로는 각 프로세스가 원하는 방식으로 자신의 사용자 가상 메모리를 배치할 수 있습니다. 하지만 실제로는 사용자 가상 메모리가 다음과 같이 배치됩니다.

```
USER_STACK +----------------------------------+
           |             user stack           |
           |                 |                |
           |                 |                |
           |                 V                |
           |           grows downward         |
           |                                  |
           |                                  |
           |                                  |
           |                                  |
           |           grows upward           |
           |                 ^                |
           |                 |                |
           |                 |                |
           +----------------------------------+
           | uninitialized data segment (BSS) |
           +----------------------------------+
           |     initialized data segment     |
           +----------------------------------+
           |            code segment          |
 0x400000  +----------------------------------+
           |                                  |
           |                                  |
           |                                  |
           |                                  |
           |                                  |
       0   +----------------------------------+
```

이번 프로젝트에서는 사용자 스택 크기가 고정되어 있지만, project 3에서는 스택이 커질 수 있도록 허용하게 됩니다. 전통적으로 초기화되지 않은 데이터 세그먼트의 크기는 시스템 콜로 조정할 수 있지만, 이것까지 구현할 필요는 없습니다.

Pintos의 코드 세그먼트는 사용자 가상 주소 `0x400000`에서 시작하며, 이는 주소 공간의 맨 아래에서 약 128 MB 위입니다. 이 값은 ubuntu에서 흔히 쓰이던 값이며, 특별히 깊은 의미는 없습니다.

링커는 각 프로그램 세그먼트의 이름과 위치를 알려 주는 "linker script"의 지시에 따라 사용자 프로그램의 메모리 레이아웃을 설정합니다. 링커 스크립트에 대해 더 알고 싶다면 `info ld`로 볼 수 있는 링커 매뉴얼의 "Scripts" 장을 읽어 보세요.

특정 실행 파일의 레이아웃을 보려면 `-p` 옵션과 함께 objdump를 실행하세요.

### 사용자 메모리 접근하기

시스템 콜의 일부로 커널은 종종 사용자 프로그램이 제공한 포인터를 통해 메모리에 접근해야 합니다. 이때 사용자가 null 포인터, 매핑되지 않은 가상 메모리를 가리키는 포인터, 혹은 커널 가상 주소 공간(`KERN_BASE` 위쪽)을 가리키는 포인터를 넘길 수 있으므로 커널은 매우 조심해야 합니다. 이런 종류의 잘못된 포인터는 모두 커널이나 다른 실행 중 프로세스에 피해를 주지 않은 채 거부되어야 하며, 문제를 일으킨 프로세스를 종료하고 그 자원을 해제해야 합니다.

이를 올바르게 처리하는 방법은 적어도 두 가지가 있습니다. 첫 번째 방법은 사용자 제공 포인터의 유효성을 먼저 검사한 뒤 역참조하는 것입니다. 이 방법을 택한다면 `thread/mmu.c`와 `include/threads/vaddr.h`의 함수들을 살펴보세요. 사용자 메모리 접근을 처리하는 가장 단순한 방법입니다.

두 번째 방법은 사용자 포인터가 `KERN_BASE` 아래를 가리키는지만 확인한 뒤 곧바로 역참조하는 것입니다. 잘못된 사용자 포인터는 "page fault"를 일으키며, 이를 `userprog/exception.c`의 `page_fault()` 코드를 수정해서 처리할 수 있습니다. 이 기법은 프로세서의 MMU를 활용하므로 보통 더 빠르며, 실제 커널(Linux 포함)에서도 이런 방식을 사용하는 경향이 있습니다.

어느 방법을 쓰든 자원이 "leak"되지 않게 해야 합니다. 예를 들어 어떤 시스템 콜이 락을 획득했거나 `malloc()`으로 메모리를 할당했다고 가정해 봅시다. 그 뒤에 잘못된 사용자 포인터를 만나더라도, 반드시 락을 해제하거나 메모리 페이지를 반납해야 합니다. 포인터를 역참조하기 전에 유효성을 검사하는 방식을 택했다면 이는 비교적 간단합니다. 반면 잘못된 포인터가 페이지 폴트를 일으키는 방식은 메모리 접근으로부터 오류 코드를 반환할 방법이 없기 때문에 처리하기가 더 어렵습니다. 그래서 후자의 기법을 시도하고 싶은 분들을 위해, 도움이 되는 코드를 조금 제공하겠습니다.

```c
/* Reads a byte at user virtual address UADDR.
 * UADDR must be below KERN_BASE.
 * Returns the byte value if successful, -1 if a segfault
 * occurred. */
static int64_t
get_user (const uint8_t *uaddr) {
    int64_t result;
    __asm __volatile (
    "movabsq $done_get, %0\n"
    "movzbq %1, %0\n"
    "done_get:\n"
    : "=&a" (result) : "m" (*uaddr));
    return result;
}

/* Writes BYTE to user address UDST.
 * UDST must be below KERN_BASE.
 * Returns true if successful, false if a segfault occurred. */
static bool
put_user (uint8_t *udst, uint8_t byte) {
    int64_t error_code;
    __asm __volatile (
    "movabsq $done_put, %0\n"
    "movb %b2, %1\n"
    "done_put:\n"
    : "=&a" (error_code), "=m" (*udst) : "q" (byte));
    return error_code != -1;
}
```

이 함수들은 모두 사용자 주소가 이미 `KERN_BASE` 아래에 있음이 검증되었다고 가정합니다. 또한 커널에서 페이지 폴트가 발생했을 때 단지 rax를 -1로 설정하고, 이전 값을 `%rip`에 복사하도록 `page_fault()`를 수정해 두었다고 가정합니다.
