# 04_system_call

### 시스템 콜

**시스템 콜 인프라를 구현하세요.**

`userprog/syscall.c`에서 시스템 콜 핸들러를 구현하세요. 우리가 제공한 skeleton 구현은 시스템 콜을 "처리"하는 대신 프로세스를 종료합니다. 여기서 시스템 콜 번호를 가져오고, 이어서 시스템 콜 인자들을 읽고, 적절한 동작을 수행하도록 만들어야 합니다.

#### 시스템 콜 세부 사항

첫 번째 프로젝트에서는 운영체제가 사용자 프로그램으로부터 제어권을 되찾는 한 가지 방법, 즉 타이머와 I/O 장치가 발생시키는 인터럽트를 이미 다뤘습니다. 이것들은 CPU 외부의 개체가 발생시키므로 "외부" 인터럽트입니다.

운영체제는 소프트웨어 예외도 다룹니다. 이것은 프로그램 코드 안에서 발생하는 이벤트입니다. 페이지 폴트나 0으로 나누기 같은 오류도 여기에 속합니다. 그리고 예외는 사용자 프로그램이 운영체제에 서비스("시스템 콜")를 요청하는 수단이기도 합니다.

전통적인 x86 아키텍처에서는 시스템 콜을 다른 소프트웨어 예외와 같은 방식으로 처리했습니다. 하지만 x86-64에서는 제조사가 시스템 콜을 위한 특별한 명령인 `syscall`을 도입했습니다. 이를 사용하면 시스템 콜 핸들러를 더 빠르게 호출할 수 있습니다.

오늘날 x86-64에서 시스템 콜을 호출하는 가장 일반적인 방법은 `syscall` 명령입니다. Pintos에서도 사용자 프로그램은 `syscall`을 사용해 시스템 콜을 호출합니다. 시스템 콜 번호와 추가 인자들은 `syscall` 명령을 실행하기 전에 일반 호출 규약에 따라 레지스터에 설정되어 있어야 하지만, 두 가지 예외가 있습니다.

- `%rax`에는 시스템 콜 번호가 들어갑니다.
- 네 번째 인자는 `%rcx`가 아니라 `%r10`입니다.

따라서 시스템 콜 핸들러 `syscall_handler()`가 제어를 넘겨받을 때 시스템 콜 번호는 rax에 들어 있고, 인자는 `%rdi`, `%rsi`, `%rdx`, `%r10`, `%r8`, `%r9` 순서로 전달됩니다.

호출자의 레지스터 값은 핸들러에 전달되는 `struct intr_frame`을 통해 접근할 수 있습니다. (`struct intr_frame`은 커널 스택 위에 있습니다.)

x86-64 호출 규약에서 함수 반환값은 RAX 레지스터에 두어야 합니다. 반환값이 있는 시스템 콜은 `struct intr_frame`의 `rax` 멤버를 수정해 이를 구현할 수 있습니다.

#### 다음 시스템 콜을 구현하세요.

아래에 나온 프로토타입은 `include/lib/user/syscall.h`를 include한 사용자 프로그램에서 보게 되는 형태입니다. (`include/lib/user` 아래의 이 헤더와 다른 모든 헤더는 사용자 프로그램 전용입니다.) 각 시스템 콜 번호는 `include/lib/syscall-nr.h`에 정의되어 있습니다.

---

```c
void halt (void);
```

> `power_off()`(`src/include/threads/init.h`에 선언됨)를 호출해 Pintos를 종료합니다. 이렇게 종료하면 가능한 deadlock 상황 등에 대한 정보를 잃게 되므로, 이 시스템 콜은 거의 사용하지 않아야 합니다.

---

```c
void exit (int status);
```

> 현재 사용자 프로그램을 종료하고, `status`를 커널에 반환합니다. 부모 프로세스가 이 프로세스를 `wait`한다면(아래 참고), 이 값이 반환됩니다. 관례적으로 `status`가 `0`이면 성공이고, 0이 아닌 값은 오류를 의미합니다.

---

```c
pid_t fork (const char *thread_name);
```

> 현재 프로세스를 복제한 새 프로세스를 THREAD_NAME이라는 이름으로 생성합니다. callee-saved 레지스터인 `%RBX`, `%RSP`, `%RBP`, `%R12` - `%R15`를 제외한 나머지 레지스터 값은 복제할 필요가 없습니다. 성공하면 자식 프로세스의 pid를 반환해야 하며, 실패하면 유효한 pid가 아니어야 합니다. 자식 프로세스에서는 반환값이 `0`이어야 합니다. 자식은 파일 디스크립터와 가상 메모리 공간을 포함한 자원을 **복제된 상태로** 가져야 합니다. 부모 프로세스는 자식 프로세스가 복제를 성공했는지 알기 전까지 `fork`에서 반환하면 안 됩니다. 즉, 자식 프로세스가 자원 복제에 실패하면 부모의 `fork()` 호출은 `TID_ERROR`를 반환해야 합니다.
>
> 템플릿은 `threads/mmu.c`의 `pml4_for_each()`를 사용해 관련 pagetable 구조를 포함한 사용자 메모리 공간 전체를 복사하지만, 전달된 `pte_for_each_func`의 비어 있는 부분은 여러분이 채워야 합니다([virtual address](../06_appendix/04_virtual_address.md) 참고).

---

```c
int exec (const char *cmd_line);
```

> 현재 프로세스를 `cmd_line`에 지정된 이름의 실행 파일로 바꾸고, 주어진 인자도 함께 전달합니다. 성공하면 이 함수는 절대 반환하지 않습니다. 반대로 어떤 이유로든 프로그램을 로드하거나 실행할 수 없으면, 프로세스는 종료 상태 `-1`로 *종료됩니다*. 이 함수는 `exec`를 호출한 스레드의 이름을 바꾸지 않습니다. 또한 `exec` 호출 전후로 파일 디스크립터는 열린 상태를 유지한다는 점에 유의하세요.

---

```c
int wait (pid_t pid);
```

> 자식 프로세스 `pid`를 기다리고 그 종료 상태를 가져옵니다. `pid`가 아직 살아 있으면 종료할 때까지 기다립니다. 그다음 `pid`가 `exit`에 전달한 상태값을 반환합니다. `pid`가 `exit()`를 호출하지 않고 커널에 의해 종료되었다면(예: 예외로 인해 kill된 경우), `wait(pid)`는 반드시 `-1`을 반환해야 합니다. 부모 프로세스가 `wait`를 호출하기 전에 자식 프로세스가 이미 종료된 경우에도 기다리는 것은 완전히 합법적입니다. 하지만 그 경우에도 커널은 부모가 자식의 종료 상태를 가져오거나, 자식이 커널에 의해 종료되었다는 사실을 알 수 있게 해야 합니다.
>
> 다음 조건 중 하나라도 참이면 `wait`는 즉시 실패하고 `-1`을 반환해야 합니다.
>
> - `pid`가 호출 프로세스의 직계 자식을 가리키지 않습니다. 어떤 `pid`가 호출 프로세스의 직계 자식인 것은, 그리고 오직 그 경우에만, 호출 프로세스가 성공적인 `fork` 호출의 반환값으로 그 `pid`를 받았을 때입니다. 자식은 상속되지 않는다는 점에 유의하세요. A가 자식 B를 만들고 B가 자식 프로세스 C를 만들었다면, B가 죽어 있더라도 A는 C를 기다릴 수 없습니다. 프로세스 A가 `wait(C)`를 호출하면 반드시 실패해야 합니다. 마찬가지로, 고아 프로세스는 부모 프로세스가 먼저 종료하더라도 새로운 부모에게 할당되지 않습니다.
> - `wait`를 호출한 프로세스가 이미 `pid`에 대해 `wait`를 한 적이 있습니다. 즉, 한 프로세스는 어떤 자식 하나에 대해 최대 한 번만 `wait`할 수 있습니다.

프로세스는 자식을 얼마든지 만들 수 있고, 어떤 순서로든 `wait`할 수 있으며, 일부 혹은 전체 자식을 기다리지 않은 채 종료할 수도 있습니다. 설계할 때는 `wait`가 일어날 수 있는 모든 경우를 고려해야 합니다. 어떤 자식이 부모보다 먼저 종료하든 나중에 종료하든, 그리고 부모가 그 자식을 기다리든 기다리지 않든 상관없이, 해당 프로세스의 모든 자원(`struct thread` 포함)은 반드시 해제되어야 합니다.

**초기 프로세스가 종료할 때까지 Pintos가 종료되지 않도록 반드시 보장해야 합니다.**  제공된 Pintos 코드는 `main()`(`threads/init.c`)에서 `process_wait()`(`userprog/process.c`)를 호출하는 방식으로 이를 시도합니다. 함수 상단 주석에 맞게 `process_wait()`를 구현한 뒤, 그 위에 wait 시스템 콜을 구현하는 방식을 권장합니다.

이 시스템 콜은 나머지 어떤 시스템 콜보다도 구현량이 훨씬 많습니다.

---

```c
bool create (const char *file, unsigned initial_size);
```

> `file`이라는 새 파일을 만들고, 초기 크기를 `initial_size`바이트로 설정합니다. 성공하면 true, 아니면 false를 반환합니다. 새 파일을 만드는 것만으로는 파일이 열리지 않습니다. 새 파일을 여는 것은 별도의 동작이며, 이를 위해서는 `open` 시스템 콜이 필요합니다.

---

```c
bool remove (const char *file);
```

> `file`이라는 파일을 삭제합니다. 성공하면 true, 아니면 false를 반환합니다. 파일은 열려 있든 닫혀 있든 삭제할 수 있으며, 열린 파일을 삭제해도 그 파일이 자동으로 닫히지는 않습니다. 자세한 내용은 [FAQ](08_faq.md)의 "열린 파일이 제거되면 어떻게 됩니까?"를 참고하세요.

---

```c
int open (const char *file);
```

> `file`이라는 파일을 엽니다. 성공하면 "file descriptor"(fd)라고 부르는 음이 아닌 정수 핸들을 반환하고, 파일을 열 수 없으면 `-1`을 반환합니다. 번호가 0과 1인 파일 디스크립터는 콘솔용으로 예약되어 있습니다. fd 0(`STDIN_FILENO`)은 표준 입력이고, fd 1(`STDOUT_FILENO`)은 표준 출력입니다. `open` 시스템 콜은 이 둘 중 어느 것도 반환하지 않으며, 이 값들은 아래에서 명시적으로 설명한 경우에만 시스템 콜 인자로서 유효합니다. 각 프로세스는 서로 독립적인 파일 디스크립터 집합을 가집니다. 파일 디스크립터는 자식 프로세스에게 상속됩니다. 하나의 파일을 한 프로세스가 여러 번 열든, 여러 프로세스가 각각 열든, `open`을 호출할 때마다 새로운 파일 디스크립터가 반환됩니다. 같은 파일을 가리키는 서로 다른 파일 디스크립터는 `close` 호출로 각각 독립적으로 닫히며, 파일 위치도 공유하지 않습니다. **extra를 구현할 때는 0부터 시작하는 정수를 반환하는 Linux 방식을 따르세요.**

---

```c
int filesize (int fd);
```

> `fd`로 열려 있는 파일의 크기를 바이트 단위로 반환합니다.

---

```c
int read (int fd, void *buffer, unsigned size);
```

> `fd`로 열려 있는 파일에서 `size`바이트를 읽어 `buffer`에 저장합니다. 실제로 읽은 바이트 수를 반환합니다(파일 끝이면 `0`). 파일 끝 이외의 이유로 읽을 수 없으면 `-1`을 반환합니다. `fd`가 0이면 `input_getc()`를 사용해 키보드에서 읽습니다.

---

```c
int write (int fd, const void *buffer, unsigned size);
```

> 열린 파일 `fd`에 `buffer`에서 `size`바이트를 씁니다. 실제로 기록한 바이트 수를 반환하며, 일부 바이트를 쓸 수 없으면 이 값은 `size`보다 작을 수 있습니다. 보통 파일 끝을 넘어 쓰면 파일이 늘어나지만, 기본 파일 시스템은 파일 확장을 구현하지 않습니다. 따라서 예상되는 동작은 파일 끝까지 가능한 만큼만 기록하고 실제로 쓴 바이트 수를 반환하거나, 아무 바이트도 쓸 수 없으면 `0`을 반환하는 것입니다. `fd`가 1이면 콘솔에 씁니다. 콘솔에 쓸 때는 size가 수백 바이트를 넘지 않는 한 `putbuf()` 한 번으로 buffer 전체를 써야 합니다(더 큰 버퍼를 나누어 쓰는 것은 합리적입니다). 그렇지 않으면 여러 프로세스가 출력한 텍스트 줄이 콘솔에서 뒤섞여, 사람도 채점 스크립트도 모두 혼란스러워질 수 있습니다.

---

```c
void seek (int fd, unsigned position);
```

> 열린 파일 `fd`에서 다음에 읽거나 쓸 바이트 위치를 `position`으로 바꿉니다. 이 값은 파일 시작으로부터의 바이트 오프셋입니다(즉, `position`이 `0`이면 파일의 시작입니다). 현재 파일 끝을 넘어 seek하는 것은 오류가 아닙니다. 이후 `read`는 파일 끝을 의미하는 0바이트를 읽게 됩니다. 이후 `write`는 파일을 확장하며, 쓰지 않은 공백은 0으로 채웁니다. (하지만 Pintos의 파일은 project 4가 끝날 때까지 길이가 고정되어 있으므로, 파일 끝 이후의 `write`는 오류를 반환합니다.) 이러한 의미론은 파일 시스템 쪽에 이미 구현되어 있으므로, 시스템 콜 구현에서 특별한 추가 작업은 필요하지 않습니다.

---

```c
unsigned tell (int fd);
```

> 열린 파일 `fd`에서 다음에 읽거나 쓸 바이트의 위치를 파일 시작으로부터의 바이트 오프셋으로 반환합니다.

---

```c
void close (int fd);
```

> 파일 디스크립터 `fd`를 닫습니다. 프로세스가 종료되거나 강제 종료될 때는, 열려 있는 각 파일 디스크립터에 대해 이 함수를 호출한 것처럼 모든 파일 디스크립터가 암묵적으로 닫힙니다.

이 파일에는 다른 시스템 콜들도 정의되어 있습니다. 지금은 무시해도 됩니다. 일부는 project 3에서, 나머지는 project 4에서 구현하게 되므로, 시스템을 확장 가능하게 설계해 두어야 합니다.

사용자 프로세스 수와 상관없이 동시에 시스템 콜을 호출할 수 있도록 시스템 콜을 동기화해야 합니다. 특히 `filesys` 디렉터리에 제공된 파일 시스템 코드는 여러 스레드가 동시에 호출해도 안전하지 않습니다. 따라서 시스템 콜 구현은 파일 시스템 코드를 critical section으로 취급해야 합니다. `process_exec()` 역시 파일에 접근한다는 점을 잊지 마세요. 현재로서는 `filesys` 디렉터리 안의 코드를 수정하지 않는 것을 권장합니다.

각 시스템 콜에 대해 사용자 수준 함수도 `lib/user/syscall.c`에 제공되어 있습니다. 이 함수들은 사용자 프로세스가 C 프로그램에서 시스템 콜을 호출할 수 있는 방법을 제공합니다. 각각은 약간의 inline assembly로 시스템 콜을 호출하고, 필요하다면 시스템 콜의 반환값도 돌려줍니다.

이 부분이 끝나면, 그리고 그 이후에도, Pintos는 어떤 경우에도 무너지지 않아야 합니다. 사용자 프로그램이 할 수 있는 어떤 일도 OS를 크래시시키거나, panic을 일으키거나, assertion을 깨뜨리거나, 그 밖의 오동작을 유발해서는 안 됩니다. 이 점은 아무리 강조해도 지나치지 않습니다. 우리의 테스트는 매우 많은 방식으로 여러분의 시스템 콜을 깨뜨리려 할 것입니다. 모든 corner case를 스스로 떠올리고 처리해야 합니다. 사용자 프로그램이 OS를 멈추게 할 수 있는 유일한 방법은 halt 시스템 콜을 호출하는 것뿐이어야 합니다.

시스템 콜에 잘못된 인자가 전달되었을 때 허용되는 선택지는 다음과 같습니다. 반환값이 있는 시스템 콜이라면 오류 값을 반환하거나, 정의되지 않은 값을 반환하거나, 혹은 프로세스를 종료할 수 있습니다.
