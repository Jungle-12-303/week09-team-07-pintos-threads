# 03_priority_scheduling

### 우선순위 스케줄링

**Pintos에 우선순위 스케줄링과 우선순위 기부(priority donation)를 구현하세요.**

현재 실행 중인 스레드보다 더 높은 우선순위를 가진 스레드가 ready list에 추가되면, 현재 스레드는 즉시 프로세서를 새 스레드에게 양보해야 합니다. 마찬가지로 스레드들이 락, 세마포어, 조건 변수를 기다리고 있을 때는 가장 높은 우선순위의 대기 스레드를 먼저 깨워야 합니다. 스레드는 언제든 자신의 우선순위를 올리거나 낮출 수 있지만, 우선순위를 낮춘 결과 더 이상 가장 높은 우선순위를 가지지 않게 되면 즉시 CPU를 양보해야 합니다.

스레드 우선순위는 `PRI_MIN (0)`부터 `PRI_MAX (63)`까지입니다. 숫자가 낮을수록 우선순위가 낮으므로 우선순위 0이 가장 낮고 우선순위 63이 가장 높습니다. 초기 스레드 우선순위는 `thread_create()`에 인자로 전달됩니다. 다른 우선순위를 선택할 특별한 이유가 없다면 `PRI_DEFAULT (31)`을 사용하세요. `PRI_` 매크로는 `threads/thread.h`에 정의되어 있으며, 그 값을 변경해서는 안 됩니다.

우선순위 스케줄링의 한 가지 문제는 "우선순위 역전(priority inversion)"입니다. 각각 높은, 중간, 낮은 우선순위의 스레드 *H*, *M*, *L*을 생각해 봅시다. *H*가 *L*을 기다려야 하고(예를 들어 *L*이 들고 있는 락 때문이라면), *M*이 ready list에 있으면 *H*는 CPU를 절대 얻지 못합니다. 낮은 우선순위 스레드가 CPU 시간을 전혀 받지 못하기 때문입니다. 이 문제를 부분적으로 해결하는 방법은, *L*이 락을 들고 있는 동안 *H*가 자기 우선순위를 *L*에게 "기부"하고, *L*이 락을 해제하면(그래서 *H*가 락을 획득하면) 그 기부를 회수하는 것입니다.

우선순위 기부가 필요한 모든 상황을 고려해 구현해야 합니다. 여러 우선순위가 하나의 스레드에 기부되는 경우도 처리해야 합니다. 중첩 기부도 처리해야 합니다. 예를 들어 *H*가 *M*이 들고 있는 락을 기다리고 있고, *M*은 다시 *L*이 들고 있는 락을 기다리고 있다면 *M*과 *L* 모두 *H*의 우선순위로 올라가야 합니다. 필요하다면 중첩된 우선순위 기부 깊이에 8단계 같은 합리적인 제한을 둘 수 있습니다.

락에 대해서는 우선순위 기부를 구현해야 합니다. 다른 Pintos 동기화 구성요소에 대해서는 우선순위 기부를 구현할 필요가 없습니다. 그러나 우선순위 스케줄링 자체는 모든 경우에 구현해야 합니다.

마지막으로 스레드가 자신의 우선순위를 조회하고 수정할 수 있게 하는 다음 함수들을 구현하세요. 이 함수들의 스켈레톤은 `threads/thread.c`에 제공되어 있습니다.

---

```c
void thread_set_priority (int new_priority);
```

> 현재 스레드의 우선순위를 새 우선순위로 설정합니다. 현재 스레드가 더 이상 가장 높은 우선순위를 가지지 않으면 yield합니다.

---

```c
int thread_get_priority (void);
```

> 현재 스레드의 우선순위를 반환합니다. 우선순위 기부가 존재하는 경우에는 더 높은 쪽의 우선순위(기부된 우선순위)를 반환합니다.

한 스레드가 다른 스레드의 우선순위를 직접 수정할 수 있게 하는 인터페이스는 제공할 필요가 없습니다.

우선순위 스케줄러는 이후 프로젝트에서는 사용되지 않습니다.
