# 04_advanced_scheduler

### 고급 스케줄러

**시스템에서 실행되는 작업의 평균 응답 시간을 줄이기 위해 **​**[4.4BSD 스케줄러](#44bsd-scheduler)**​**와 비슷한 다단계 피드백 큐 스케줄러를 구현하세요.**

우선순위 스케줄러와 마찬가지로 고급 스케줄러도 우선순위를 기준으로 실행할 스레드를 선택합니다. 하지만 고급 스케줄러는 우선순위 기부를 수행하지 않습니다. 따라서 고급 스케줄러 작업을 시작하기 전에, 적어도 우선순위 기부를 제외한 우선순위 스케줄러는 동작하도록 만들어 두기를 권장합니다.

Pintos 시작 시 스케줄링 알고리즘 정책을 선택할 수 있도록 코드를 작성해야 합니다. 기본적으로는 우선순위 스케줄러가 활성화되어 있어야 하지만, `-mlfqs` 커널 옵션으로 4.4BSD 스케줄러를 선택할 수 있어야 합니다. 이 옵션을 전달하면 `threads/thread.h`에 선언된 `thread_mlfqs`가 `main()` 초반에 일어나는 `parse_options()`의 옵션 파싱 과정에서 `true`로 설정됩니다.

4.4BSD 스케줄러가 활성화되면 스레드는 더 이상 자기 우선순위를 직접 제어하지 않습니다. `thread_create()`의 우선순위 인자는 무시되어야 하고, `thread_set_priority()` 호출도 마찬가지입니다. `thread_get_priority()`는 스케줄러가 설정한 현재 우선순위를 반환해야 합니다. 고급 스케줄러는 이후 프로젝트에서는 사용되지 않습니다.

### 4.4BSD 스케줄러

범용 스케줄러의 목표는 스레드마다 서로 다른 스케줄링 요구를 균형 있게 맞추는 것입니다. I/O를 많이 수행하는 스레드는 입력 장치와 출력 장치가 바쁘게 유지되도록 빠른 응답 시간이 필요하지만, CPU 시간은 많이 필요하지 않습니다. 반면 계산 중심 스레드는 작업을 끝내기 위해 많은 CPU 시간을 받아야 하지만, 빠른 응답 시간은 필요하지 않습니다. 그 밖의 스레드는 이 둘의 중간 어디쯤에 있으며, I/O 구간과 계산 구간이 번갈아 나타나기 때문에 시간에 따라 요구가 달라집니다. 잘 설계된 스케줄러는 이런 요구를 동시에 만족시키는 경우가 많습니다.

프로젝트 1에서는 이 부록에 설명된 스케줄러를 구현해야 합니다. 우리의 스케줄러는 [McKusick]에 설명된 것과 닮아 있는데, 이것은 다단계 피드백 큐 스케줄러의 한 예입니다. 이런 종류의 스케줄러는 실행 가능한 스레드를 담은 여러 개의 큐를 유지하며, 각 큐에는 서로 다른 우선순위의 스레드가 들어 있습니다. 어느 시점이든 스케줄러는 비어 있지 않은 가장 높은 우선순위 큐에서 스레드 하나를 선택합니다. 가장 높은 우선순위 큐에 여러 스레드가 있으면, 이들은 "라운드 로빈" 순서로 실행됩니다.

스케줄러의 여러 동작은 일정한 수의 타이머 틱 뒤에 데이터를 갱신해야 합니다. 모든 경우에 이런 갱신은 일반 커널 스레드가 실행될 기회를 갖기 전에 일어나야 합니다. 그래야 커널 스레드가 막 증가한 `timer_ticks()` 값은 보지만 스케줄러 데이터 값은 예전 것인 상태를 보는 일이 없습니다.

4.4BSD 스케줄러에는 우선순위 기부가 없습니다.

#### Nice 값

스레드 우선순위는 아래에 제시된 공식을 사용해 스케줄러가 동적으로 결정합니다. 하지만 각 스레드에는 다른 스레드에 얼마나 "친절하게" CPU를 양보할지를 나타내는 정수형 nice 값도 있습니다. nice 값이 0이면 스레드 우선순위에 영향을 주지 않습니다. 양수 nice 값은 최대 20까지 가능하며, 스레드의 우선순위를 낮추고 원래 받을 수 있었던 CPU 시간의 일부를 포기하게 만듭니다. 반대로 음수 nice 값은 최소 -20까지 가능하며, 다른 스레드로부터 CPU 시간을 빼앗는 경향이 있습니다.

초기 스레드는 nice 값 0으로 시작합니다. 다른 스레드는 부모 스레드의 nice 값을 상속받아 시작합니다. 아래에 설명된 함수들을 구현해야 하며, 이 함수들은 테스트 프로그램에서 사용됩니다. 이 함수들의 스켈레톤 정의는 `threads/thread.c`에 제공되어 있습니다.

```c
int thread_get_nice (void);
```

> 현재 스레드의 nice 값을 반환합니다.

```c
int thread_set_nice (int nice);
```

> 현재 스레드의 nice 값을 새 nice 값으로 설정하고, 새 값에 기반해 스레드의 우선순위를 다시 계산합니다
> ([우선순위 계산](#calculating-priority) 참고). 실행 중인 스레드가 더 이상 가장 높은 우선순위를 가지지 않게 되면 yield합니다.

#### 우선순위 계산

우리의 스케줄러는 64개의 우선순위, 즉 64개의 준비 큐(ready queue)를 가지며, 번호는 0 `(PRI_MIN)`부터 `63 (PRI_MAX)`까지입니다. 숫자가 낮을수록 우선순위가 낮으므로 우선순위 0이 가장 낮고 우선순위 63이 가장 높습니다. 스레드 우선순위는 스레드 초기화 시 처음 계산됩니다. 그 후에는 모든 스레드에 대해 매 네 번째 클록 틱마다 다시 계산됩니다. 두 경우 모두 다음 공식을 사용합니다.

> priority = PRI_MAX - (recent_cpu / 4) - (nice * 2),

여기서 `recent_cpu`는 스레드가 최근에 사용한 CPU 시간의 추정치이고(아래 참고), nice는 해당 스레드의 nice 값입니다. 결과는 소수 부분을 버려 정수로 만들어야 합니다(truncated). `recent_cpu`와 nice에 대한 계수 1/4와 2는 실제로 잘 동작하는 것으로 알려져 있지만, 더 깊은 의미가 있는 것은 아닙니다. 계산된 priority는 항상 `PRI_MIN`에서 `PRI_MAX` 사이의 유효 범위로 조정됩니다.

이 공식은 최근에 CPU 시간을 받은 스레드가 다음에 스케줄러가 실행될 때 CPU를 다시 배정받을 우선순위를 낮춰 줍니다. 이는 기아(starvation)를 막는 핵심입니다. 최근에 CPU 시간을 전혀 받지 못한 스레드는 `recent_cpu`가 0이 되므로, nice 값이 지나치게 높지 않은 한 곧 CPU 시간을 받을 수 있게 됩니다.

#### `recent_cpu` 계산

우리는 `recent_cpu`가 각 프로세스가 "최근에" 얼마나 많은 CPU 시간을 받았는지를 측정하기를 원합니다. 더 나아가, 더 최근의 CPU 시간일수록 오래된 CPU 시간보다 더 큰 가중치를 받아야 합니다. 한 가지 접근은 최근 n초 각각에 받은 CPU 시간을 추적하는 n개의 원소를 가진 배열을 사용하는 것입니다. 하지만 이 접근은 스레드마다 O(n)의 공간이 필요하고, 새로운 가중 평균을 계산할 때마다 O(n)의 시간이 듭니다.

대신 지수 가중 이동 평균(exponentially weighted moving average)을 사용합니다. 일반 형태는 다음과 같습니다.

> x(0) = f(0),
>
> x(t) = ax(t − 1) + (1 − a)f(t),
>
> a = k/(k + 1),

여기서 `x(t)`는 정수 시각 `t ≥ 0`에서의 이동 평균이고, `f(t)`는 평균을 내고 있는 함수이며, `k > 0`는 감쇠 속도를 제어합니다. 이 공식을 몇 단계 전개하면 다음과 같습니다.

> x(1) = f(1),
>
> x(2) = af(1) + f(2),
>
> ...
>
> x(5) = a<sup>4 * f(1) + a</sup>3 * f(2) + a^2 * f(3) + a * f(4) + f(5).

`f(t)`의 값은 시각 `t`에서 가중치 1을 가지며, 시각 `t + 1`에서는 `a`, `t + 2`에서는 `a^2`의 가중치를 갖는 식으로 이어집니다. 반대로 `x(t)`를 `k`와 관련지어 볼 수도 있습니다. `f(t)`는 시각 `t + k`에서 대략 `1/e`의 가중치를 가지며, `t + 2k`에서는 대략 `1/(e^2)`의 가중치를 가집니다. 반대 방향으로 보면 `f(t)`는 시각 `t + log_a(w)`에서 가중치 `w`로 감소합니다.

`recent_cpu`의 초기값은 처음 생성되는 스레드에서는 0이고, 그 밖의 새 스레드에서는 부모의 값입니다. 타이머 인터럽트가 발생할 때마다, idle 스레드가 실행 중인 경우를 제외하고 실행 중인 스레드의 `recent_cpu`만 1 증가합니다. 또한 초당 한 번씩 다음 공식을 사용해 모든 스레드(실행 중인지, ready 상태인지, blocked 상태인지와 무관하게)의 `recent_cpu` 값을 다시 계산합니다.

> recent_cpu = (2 * load_avg)/(2 * load_avg + 1) * recent_cpu + nice

여기서 `load_avg`는 실행 준비가 된 스레드 수의 이동 평균입니다(아래 참고). `load_avg`가 1이라면, 즉 평균적으로 하나의 스레드가 CPU를 두고 경쟁하고 있다는 뜻이라면 현재 `recent_cpu` 값의 가중치는 `log_(2/3) .1 ≈ 6`초 안에 `.1`로 감소합니다. `load_avg`가 2라면 `.1`로 감소하는 데 `log_(3/4) .1 ≈ 8`초가 걸립니다. 즉 `recent_cpu`는 스레드가 "최근에" 얼마나 많은 CPU 시간을 받았는지를 추정하며, 감쇠 속도는 CPU를 두고 경쟁하는 스레드 수에 반비례합니다.

일부 테스트가 두는 가정 때문에 `recent_cpu`의 재계산은 시스템 틱 카운터가 정확히 1초의 배수에 도달했을 때, 즉 `timer_ticks () % TIMER_FREQ == 0`일 때에만 수행되어야 하며 다른 시점에는 수행되면 안 됩니다.

nice 값이 음수인 스레드의 경우 `recent_cpu` 값도 음수가 될 수 있습니다.
음수 `recent_cpu`를 0으로 고정하지 마세요.

이 공식에서는 계산 순서를 고민할 필요가 있을 수 있습니다. 우리는 먼저 `recent_cpu`의 계수를 계산한 뒤 곱하는 것을 권장합니다. 일부 학생들은 `load_avg`와 `recent_cpu`를 바로 곱하면 overflow가 발생할 수 있다고 보고했습니다.

`thread_get_recent_cpu()`를 구현해야 합니다. `threads/thread.c`에 스켈레톤이 제공되어 있습니다.

```c
int thread_get_recent_cpu (void);
```

> 현재 스레드의 `recent_cpu` 값에 100을 곱한 뒤 가장 가까운 정수로 반올림한 값을 반환합니다.

#### `load_avg` 계산

마지막으로 `load_avg`, 흔히 시스템 부하 평균(load average)이라고 부르는 값은 지난 1분 동안 실행 준비가 된 스레드 수의 평균을 추정합니다. `recent_cpu`와 마찬가지로 지수 가중 이동 평균입니다. 그러나 우선순위와 `recent_cpu`와는 달리 `load_avg`는 스레드별 값이 아니라 시스템 전체 값입니다. 시스템 부팅 시에는 0으로 초기화됩니다. 그 이후에는 초당 한 번씩 다음 공식에 따라 갱신됩니다.

> load_avg = (59/60) * load_avg + (1/60) * ready_threads,

여기서 ready threads는 갱신 시점에 실행 중이거나 실행 준비가 된 스레드의 수입니다(idle 스레드는 제외).

일부 테스트가 두는 가정 때문에 `load_avg` 역시 시스템 틱 카운터가 정확히 1초의 배수에 도달했을 때, 즉 `timer_ticks () % TIMER_FREQ == 0`일 때에만 갱신되어야 하며 다른 시점에는 갱신되면 안 됩니다.
`thread_get_load_avg()`를 구현해야 하며, `threads/thread.c`에 스켈레톤이 제공되어 있습니다.

```c
int thread_get_load_avg (void)
```

> 현재 시스템 `load_avg` 값에 100을 곱한 뒤 가장 가까운 정수로 반올림한 값을 반환합니다.

#### 요약

다음 공식들은 스케줄러 구현에 필요한 계산을 요약한 것입니다. 이것만으로 스케줄러 요구사항이 완전히 설명되지는 않습니다.

모든 스레드는 -20에서 20 사이의 nice 값을 직접 제어합니다.
각 스레드는 또한 0 (`PRI_MIN`)부터 63 (`PRI_MAX`) 사이의 priority를 가지며, 이 값은 매 네 번째 틱마다 다음 공식으로 다시 계산됩니다.

> priority = PRI_MAX - (recent_cpu / 4) - (nice * 2)

`recent_cpu`는 스레드가 "최근에" 받은 CPU 시간의 양을 측정합니다.
매 타이머 틱마다 실행 중인 스레드의 `recent_cpu`는 1 증가합니다. 초당 한 번씩 모든 스레드의 `recent_cpu`는 다음과 같이 갱신됩니다.

> recent_cpu = (2 * load_avg) / (2 * load_avg + 1) * recent_cpu + nice

`load_avg`는 지난 1분 동안 실행 준비가 된 스레드 수의 평균을 추정합니다. 부팅 시 0으로 초기화되며, 그 후에는 초당 한 번씩 다음과 같이 다시 계산됩니다.

> load_avg = (59/60) * load_avg + (1/60) * ready_threads

여기서 `ready_threads`는 갱신 시점에 실행 중이거나 실행 준비가 된 스레드의 수입니다(idle 스레드는 제외).

#### 고정소수점 실수 연산

위 공식들에서 `priority`, `nice`, `ready_threads`는 정수이지만, `recent_cpu`와 `load_avg`는 실수입니다. 안타깝게도 Pintos는 커널에서 부동소수점 연산을 지원하지 않습니다. 그렇게 하면 커널이 더 복잡해지고 느려지기 때문입니다. 실제 커널도 같은 이유로 이런 제한을 두는 경우가 많습니다. 따라서 실수량에 대한 계산은 정수를 사용해 모사해야 합니다. 어렵지는 않지만 많은 학생이 이 방법을 잘 모릅니다. 이 절에서는 그 기초를 설명합니다.

기본 아이디어는 정수의 가장 오른쪽 비트들을 분수를 나타내는 비트로 취급하는 것입니다. 예를 들어 부호 있는 32비트 정수의 하위 14비트를 소수부 비트로 지정하면, 정수 x는 실수 `x/(2^14)`를 나타내게 됩니다. 이를 17.14 고정소수점 표현이라고 부르는데, 소수점 앞에 17비트, 소수점 뒤에 14비트, 그리고 부호 비트 1개가 있기 때문입니다. 17.14 형식의 수는 최대 `(2^31 − 1)/(2^14) ≈ 131,071.999` 값을 표현할 수 있습니다.

`p.q` 고정소수점 형식을 사용한다고 가정하고, `f = 2^q`라고 하겠습니다. 위 정의에 따르면 정수나 실수를 `p.q` 형식으로 변환하려면 `f`를 곱하면 됩니다. 예를 들어 위의 `load_avg` 계산에 사용된 분수 `59/60`은 17.14 형식에서는 `(59/60)2^14 = 16,110`입니다. 고정소수점 값을 다시 정수로 바꾸려면 `f`로 나누면 됩니다. (C의 일반 `/` 연산자는 0을 향해 반올림합니다. 즉 양수는 아래로, 음수는 위로 반올림합니다. 가장 가까운 정수로 반올림하려면 나누기 전에 양수에는 `f / 2`를 더하고, 음수에는 그 값을 빼면 됩니다.)

고정소수점 수에 대한 많은 연산은 간단합니다. `x`와 `y`를 고정소수점 수라고 하고, `n`을 정수라고 합시다. 그러면 `x`와 `y`의 합은 `x + y`이고 차는 `x - y`입니다. `x`와 `n`의 합은 `x + n * f`이고, 차는 `x - n * f`, 곱은 `x * n`, 몫은 `x / n`입니다.

두 고정소수점 값을 곱할 때는 두 가지 문제가 있습니다. 첫째, 결과의 소수점 위치가 q비트만큼 너무 왼쪽으로 갑니다. `(59/60)(59/60)`은 1보다 약간 작은 값이어야 하지만, `16,111 × 16,111 = 259,564,321`은 `2^14 = 16,384`보다 훨씬 큽니다. 이를 q비트 오른쪽으로 이동하면 `259,564,321/2^14 = 15,842`, 즉 약 `0.97`이 되어 올바른 답이 됩니다. 둘째, 답 자체는 표현 가능한데 곱셈 도중 overflow가 날 수 있습니다. 예를 들어 17.14 형식에서 `64`는 `64 × 2^14 = 1,048,576`인데, 그 제곱인 `64^2 = 4,096`은 17.14 범위 안에 충분히 들어갑니다. 그러나 `1,048,576^2 = 2^40`은 부호 있는 32비트 정수의 최대값 `2^31 − 1`보다 큽니다. 간단한 해결책은 곱셈을 64비트 연산으로 수행하는 것입니다. 그러면 x와 y의 곱은 `((int64_t) x) * y / f`가 됩니다.

두 고정소수점 값을 나눌 때는 반대 문제가 생깁니다. 소수점 위치가 너무 오른쪽으로 가므로, 나누기 전에 피제수를 q비트 왼쪽으로 이동시켜 이를 바로잡습니다. 왼쪽 시프트를 하면 피제수의 상위 q비트가 버려지는데, 이것도 마찬가지로 64비트로 나눗셈을 수행하면 해결할 수 있습니다. 따라서 x를 y로 나눈 몫은 `((int64_t) x) * f / y`가 됩니다.

이 절에서는 일관되게 q비트 시프트 대신 `f`에 대한 곱셈 또는 나눗셈을 사용했습니다. 이유는 두 가지입니다. 첫째, 곱셈과 나눗셈은 C의 시프트 연산자가 가진 놀라운 연산자 우선순위 문제를 피할 수 있습니다. 둘째, 곱셈과 나눗셈은 음수 피연산자에 대해서도 정의가 분명하지만, C의 시프트 연산자는 그렇지 않습니다. 구현할 때 이 점들을 주의하세요.

다음 표는 고정소수점 연산을 C에서 어떻게 구현할 수 있는지 요약합니다. 표에서 `x`와 `y`는 고정소수점 수, `n`은 정수이며, 고정소수점 수는 `p + q = 31`인 부호 있는 `p.q` 형식이고, `f`는 `1 << q`입니다.

|연산|C|
| -----------------------------------------| ----|
|`n`을 고정소수점으로 변환|`n * f`|
|`x`를 정수로 변환(0을 향해 반올림)|`x / f`|
|`x`를 정수로 변환(가장 가까운 정수로 반올림)|`(x + f / 2) / f` if `x >= 0`|
||`(x - f / 2) / f` if `x <= 0`|
|`x`와 `y` 더하기|`x + y`|
|`x`에서 `y` 빼기|`x - y`|
|`x`와 `n` 더하기|`x + n * f`|
|`x`에서 `n` 빼기|`x - n * f`|
|`x`와 `y` 곱하기|`((int64_t) x) * y / f`|
|`x`와 `n` 곱하기|`x * n`|
|`x`를 `y`로 나누기|`((int64_t) x) * f / y`|
|`x`를 `n`으로 나누기|`x / n`|
