# 01_introduction

# 프로젝트 1: 스레드

이 과제에서는 최소한으로 동작하는 스레드 시스템을 제공합니다. 여러분의 작업은 이 시스템의 기능을 확장하여 동기화 문제를 더 잘 이해하는 것입니다. 이번 과제에서는 주로 `threads` 디렉터리에서 작업하고, 일부는 `devices` 디렉터리에서도 작업하게 됩니다. 컴파일은 `threads` 디렉터리에서 해야 합니다.
이 프로젝트 설명을 읽기 전에 [동기화](../06_appendix/02_synchronization.md) 자료를 최소한 훑어보시기 바랍니다.

## 배경

### 스레드 이해하기

첫 단계는 초기 스레드 시스템의 코드를 읽고 이해하는 것입니다. Pintos는 이미 스레드 생성과 스레드 종료, 스레드 사이를 전환하는 단순한 스케줄러, 그리고 동기화 기본 요소(세마포어, 락, 조건 변수, 최적화 배리어)를 구현해 두었습니다.

이 코드 중 일부는 다소 낯설게 느껴질 수 있습니다. 아직 기본 시스템을 컴파일하고 실행해 보지 않았다면 [소개](../05_project4_file_system/01_introduction.md)에서 설명한 대로 지금 해보시기 바랍니다.
무슨 일이 일어나는지 확인하기 위해 소스 코드 일부를 읽어보실 수 있습니다. 원한다면 거의 어디에나 `printf()` 호출을 추가한 뒤 다시 컴파일하고 실행하여 어떤 일이 어떤 순서로 일어나는지 살펴볼 수도 있습니다. 또한 디버거에서 커널을 실행해 흥미로운 지점에 브레이크포인트를 설정하고, 코드를 한 단계씩 실행하면서 데이터를 조사할 수도 있습니다.

스레드가 생성될 때는 스케줄링될 새로운 컨텍스트를 만드는 것입니다. 이 컨텍스트에서 실행할 함수를 thread_create()의 인자로 전달합니다. 스레드가 처음 스케줄되어 실행될 때는 그 함수의 시작 지점부터 해당 컨텍스트에서 실행을 시작합니다. 함수가 반환되면 스레드는 종료합니다. 따라서 각 스레드는 Pintos 내부에서 실행되는 하나의 작은 프로그램처럼 동작하며, `thread_create()`에 넘긴 함수는 `main()`과 비슷한 역할을 합니다.

어느 시점이든 정확히 하나의 스레드만 실행되고, 나머지 스레드가 있다면 모두 비활성 상태가 됩니다. 다음에 어떤 스레드를 실행할지는 스케줄러가 결정합니다. (어떤 시점에도 실행 가능한 스레드가 없다면 `idle()`에 구현된 특수한 *idle* 스레드가 실행됩니다.) 한 스레드가 다른 스레드의 작업을 기다려야 할 때는 동기화 기본 요소가 컨텍스트 스위치를 강제로 일으킬 수 있습니다.

컨텍스트 스위치의 메커니즘은 `threads/thread.c`의 `thread_launch()`에 있습니다. (이 부분을 이해할 필요는 없습니다.) 이 함수는 현재 실행 중인 스레드의 상태를 저장하고, 전환 대상 스레드의 상태를 복원합니다.

GDB 디버거를 사용해 컨텍스트 스위치를 천천히 추적해 보며 어떤 일이 일어나는지 확인해 보세요([GDB](../06_appendix/06_debugging_tools.md#gdb) 참고). 처음에는 `schedule()`에 브레이크포인트를 설정하고, 거기서부터 한 단계씩 실행해 나가면 됩니다. 각 스레드의 주소와 상태, 그리고 각 스레드의 호출 스택에 어떤 프로시저가 올라가 있는지를 반드시 추적하세요. 한 스레드가 `do_iret()`에서 `iret`를 실행하면 다른 스레드가 실행되기 시작한다는 점을 알 수 있을 것입니다.

**경고**: Pintos에서는 각 스레드에 크기가 4 kB보다 약간 작은, 작고 고정된 실행 스택이 할당됩니다. 커널은 스택 오버플로를 감지하려고 하지만 완벽하게 감지할 수는 없습니다. `int buf[1000];` 같은 큰 데이터 구조를 non-static 지역 변수로 선언하면 원인을 알 수 없는 커널 패닉 같은 이상한 문제가 생길 수 있습니다. 스택 할당 대신 사용할 수 있는 방법으로는 페이지 할당자와 블록 할당자([메모리 할당](../06_appendix/03_memory_allocation.md) 참고)가 있습니다.

### 소스 파일

다음은 `threads`와 `include/threads` 디렉터리 안의 파일을 간단히 정리한 것입니다. 대부분의 코드는 수정할 필요가 없겠지만, 이 개요가 어떤 코드를 봐야 할지 감을 잡는 데 도움이 되기를 바랍니다.

#### `threads` 코드

- `loader.S`, `loader.h`

  커널 로더입니다. PC BIOS가 메모리에 적재하는 512바이트의 코드와 데이터로 어셈블되며, 이후 디스크에서 커널을 찾아 메모리에 적재하고 `start.S`의 `bootstrap()`으로 점프합니다. 이 코드를 보거나 수정할 필요는 없을 것입니다. `start.S`는 메모리 보호에 필요한 기본 설정을 수행하고 `64bit long mode`로 진입합니다. 로더와 달리 이 코드는 실제로 커널의 일부입니다.
- `kernel.lds.S`

  커널을 링크할 때 사용하는 링커 스크립트입니다. 커널의 로드 주소를 설정하고 `start.S`가 커널 이미지의 시작 부분에 오도록 배치합니다. 다시 말해 이 코드도 보거나 수정할 필요는 없겠지만, 궁금하다면 참고할 수 있습니다.
- `init.c`, `init.h`

  `main()`을 포함한 커널 초기화 코드, 즉 커널의 `main program`입니다. 무엇이 초기화되는지 보기 위해 최소한 `main()`은 살펴보는 것이 좋습니다. 여기에 여러분만의 초기화 코드를 추가하고 싶을 수도 있습니다.
- `thread.c`, `thread.h`

  기본 스레드 지원 코드입니다. 여러분 작업의 상당 부분이 이 파일들에서 이루어질 것입니다. `thread.h`는 `struct thread`를 정의하며, 네 개 프로젝트 모두에서 이 구조체를 수정하게 될 가능성이 큽니다. 자세한 내용은 [스레드](../06_appendix/01_threads.md)를 참고하세요.
- `palloc.c`, `palloc.h`

  4 kB 페이지 단위로 시스템 메모리를 할당하는 페이지 할당자입니다. 자세한 내용은 [페이지 할당자](../06_appendix/03_memory_allocation.md#page-allocator)를 참고하세요.
- `malloc.c`, `malloc.h`

  커널용 `malloc()`과 `free()`의 단순한 구현입니다. 자세한 내용은 [블록 할당자](../06_appendix/03_memory_allocation.md#block-allocator)를 참고하세요.
- `interrupt.c`, `interrupt.h`

  기본 인터럽트 처리 코드와 인터럽트를 켜고 끄는 함수들입니다.
- `intr-stubs.S`, `intr-stubs.h`

  저수준 인터럽트 처리를 위한 어셈블리 코드입니다.
- `synch.c`, `synch.h`

  기본 동기화 기본 요소입니다. 세마포어, 락, 조건 변수, 최적화 배리어가 들어 있습니다. 네 개 프로젝트 모두에서 동기화를 위해 이를 사용해야 합니다. 자세한 내용은 [동기화](../06_appendix/02_synchronization.md)를 참고하세요.
- `mmu.c`, `mmu.h`

  x86-64 페이지 테이블 연산을 위한 함수들입니다. lab1 이후에는 이 파일을 더 자세히 보게 될 것입니다.
- `io.h`

  I/O 포트 접근을 위한 함수들입니다. 이는 대부분 여러분이 손댈 필요가 없는 `devices` 디렉터리의 소스 코드에서 사용됩니다.
- `vaddr.h`, `pte.h`

  가상 주소와 페이지 테이블 엔트리를 다루는 함수와 매크로입니다. 이 파일들은 프로젝트 3에서 더 중요해집니다. 지금은 무시해도 됩니다.
- `flags.h`

  x86-64 `flags` 레지스터의 몇몇 비트를 정의하는 매크로입니다. 아마 크게 관심이 없을 것입니다.

#### `devices` 코드

기본 스레드 커널에는 `devices` 디렉터리의 다음 파일들도 포함됩니다.

- `timer.c`, `timer.h`

  기본적으로 초당 100번 틱하는 시스템 타이머입니다. 이 프로젝트에서 이 코드를 수정하게 됩니다.
- `vga.c`, `vga.h`

  VGA 디스플레이 드라이버입니다. 화면에 텍스트를 쓰는 일을 담당합니다. 이 코드를 볼 필요는 없을 것입니다. `printf()`가 여러분 대신 VGA 디스플레이 드라이버를 호출하므로, 이 코드를 직접 호출할 이유는 거의 없습니다.
- `serial.c`, `serial.h`

  직렬 포트 드라이버입니다. 마찬가지로 `printf()`가 여러분 대신 이 코드를 호출하므로 직접 호출할 필요가 없습니다. 직렬 입력은 입력 레이어로 전달하여 처리합니다(아래 참고).
- `block.c`, `block.h`

  블록 장치를 위한 추상화 레이어입니다. 블록 장치란 고정 크기 블록의 배열로 구성된, 임의 접근이 가능한 디스크형 장치를 말합니다. 기본 상태의 Pintos는 두 종류의 블록 장치, 즉 IDE 디스크와 파티션을 지원합니다. 어떤 종류이든 블록 장치는 프로젝트 2가 되기 전까지는 실제로 사용되지 않습니다.
- `ide.c`, `ide.h`

  최대 4개의 IDE 디스크에서 섹터를 읽고 쓸 수 있게 지원합니다.
- `partition.c`, `partition.h`

  디스크의 파티션 구조를 이해하여 하나의 디스크를 여러 영역(파티션)으로 나누어 각각 독립적으로 사용할 수 있게 합니다.
- `kbd.c`, `kbd.h`

  키보드 드라이버입니다. 키 입력을 받아 입력 레이어로 전달합니다(아래 참고).
- `input.c`, `input.h`

  입력 레이어입니다. 키보드나 직렬 드라이버가 전달한 입력 문자를 큐에 넣습니다.
- `intq.c`, `intq.h`

  인터럽트 큐입니다. 커널 스레드와 인터럽트 핸들러가 모두 접근하려는 원형 큐를 관리합니다. 키보드 및 직렬 드라이버에서 사용됩니다.
- `rtc.c`, `rtc.h`

  실시간 시계 드라이버입니다. 커널이 현재 날짜와 시간을 알아낼 수 있게 합니다. 기본 설정에서는 `thread/init.c`가 난수 생성기의 초기 시드를 고를 때에만 사용됩니다.
- `speaker.c`, `speaker.h`

  PC 스피커로 소리를 낼 수 있게 해주는 드라이버입니다.
- `pit.c`, `pit.h`

  8254 Programmable Interrupt Timer를 설정하는 코드입니다. 각 장치가 PIT의 출력 채널 중 하나를 사용하기 때문에 이 코드는 `devices/timer.c`와 `devices/speaker.c` 모두에서 사용됩니다.

  `stdint.h`, `stdio.c`, `stdio.h`, `stdlib.c`, `stdlib.h`, `string.c`,
  `string.h`

  표준 C 라이브러리의 일부입니다.
- `debug.c`, `debug.h`

  디버깅을 돕는 함수와 매크로입니다. 자세한 내용은 [디버깅 도구](../06_appendix/06_debugging_tools.md)를 참고하세요.
- `random.c`, `random.h`

  의사 난수 생성기입니다. 실제 난수 값의 순서는 Pintos를 여러 번 실행해도 달라지지 않습니다.
- `round.h`

  반올림을 위한 매크로입니다.
- `syscall-nr.h`

  시스템 콜 번호입니다. 프로젝트 2 전까지는 사용되지 않습니다.
- `kernel/list.c`, `kernel/list.h`

  이중 연결 리스트 구현입니다. Pintos 코드 전반에서 사용되며, 여러분도 프로젝트 1에서 몇 군데 직접 사용하고 싶을 가능성이 큽니다. **시작하기 전에 이 코드를 훑어보기를 권장합니다(특히 헤더 파일의 주석).**
- `kernel/bitmap.c`, `kernel/bitmap.h`

  비트맵 구현입니다. 원한다면 코드에서 사용할 수 있지만, 프로젝트 1에서는 필요할 가능성이 거의 없습니다.
- `kernel/hash.c`, `kernel/hash.h`

  해시 테이블 구현입니다. 프로젝트 3에서 유용할 가능성이 큽니다.
- `kernel/console.c`, `kernel/console.h`, `kernel/stdio.h`

  `printf()`와 몇몇 다른 함수를 구현합니다.

### 동기화

적절한 동기화는 이 문제들을 해결하는 데 매우 중요합니다. 인터럽트를 끄면 어떤 동기화 문제든 쉽게 해결할 수 있습니다. 인터럽트가 꺼져 있는 동안에는 동시성이 없으므로 race condition이 발생할 수 없기 때문입니다. 그래서 모든 동기화 문제를 이런 방식으로 해결하고 싶어질 수 있지만, 그렇게 하지 마세요. 대신 대부분의 동기화 문제는 세마포어, 락, 조건 변수를 사용해 해결하십시오. 어떤 상황에서 어떤 동기화 기본 요소를 사용할 수 있는지 확신이 서지 않는다면 동기화에 대한 설명 부분([동기화](../06_appendix/02_synchronization.md))이나 `threads/synch.c`의 주석을 읽어보세요.

Pintos 프로젝트에서 인터럽트를 비활성화하는 방식이 가장 적합한 문제는 커널 스레드와 인터럽트 핸들러가 공유하는 데이터를 조율하는 경우뿐입니다. 인터럽트 핸들러는 sleep 상태로 들어갈 수 없으므로 락을 획득할 수 없습니다. 따라서 커널 스레드와 인터럽트 핸들러가 공유하는 데이터는 커널 스레드 쪽에서 인터럽트를 꺼서 보호해야 합니다.

이 프로젝트에서는 인터럽트 핸들러에서 소량의 스레드 상태만 접근하면 됩니다. 알람 시계에서는 타이머 인터럽트가 잠든 스레드를 깨워야 합니다. 고급 스케줄러에서는 타이머 인터럽트가 몇몇 전역 변수와 스레드별 변수에 접근해야 합니다. 커널 스레드에서 이 변수들에 접근할 때는 타이머 인터럽트가 간섭하지 못하도록 인터럽트를 비활성화해야 합니다.

인터럽트를 꺼야 할 때는 가능한 한 최소한의 코드에서만 그렇게 하도록 주의하십시오. 그렇지 않으면 타이머 틱이나 입력 이벤트 같은 중요한 것을 놓칠 수 있습니다. 인터럽트를 끄면 인터럽트 처리 지연도 증가하므로, 지나치면 시스템이 굼뜨게 느껴질 수 있습니다.

`synch.c`의 동기화 기본 요소 자체도 인터럽트를 비활성화하는 방식으로 구현되어 있습니다. 이곳에서 인터럽트를 끈 채 실행되는 코드의 범위를 늘려야 할 수도 있지만, 그래도 그 범위는 최소화하려고 해야 합니다.

인터럽트를 비활성화하는 것은 디버깅에도 유용할 수 있습니다. 어떤 코드 구간이 중단되지 않는다는 점을 확실히 하고 싶을 때 도움이 됩니다. 프로젝트를 제출하기 전에 디버깅 코드는 제거해야 합니다. (그냥 주석 처리만 해 두지는 마세요. 코드를 읽기 어렵게 만들 수 있기 때문입니다.)

제출물에는 busy waiting이 없어야 합니다. `thread_yield()`를 호출하는 타이트한 루프는 busy waiting의 한 형태입니다.

### 개발 제안

과거에는 많은 조가 과제를 여러 조각으로 나눈 뒤, 각 구성원이 마감 직전까지 자기 몫만 작업하고 마지막에 다시 모여 코드를 합쳐 제출하곤 했습니다. 이는 좋지 않은 생각입니다. 우리는 이런 접근을 권장하지 않습니다. 이렇게 하면 두 변경 사항이 서로 충돌해 마감 직전에 대량의 디버깅이 필요해지는 경우가 많습니다. 실제로 어떤 조는 컴파일이나 부팅조차 되지 않는 코드를 제출했고, 테스트 통과는 더더욱 하지 못했습니다.

대신 [git](https://git-scm.com/docs/gittutorial) 같은 소스 코드 관리 시스템을 사용해 팀의 변경 사항을 일찍, 그리고 자주 통합할 것을 권장합니다. 이렇게 하면 각자 작업이 끝났을 때만이 아니라 작성되는 동안에도 서로의 코드를 볼 수 있으므로 예상치 못한 문제가 생길 가능성이 줄어듭니다. 이런 시스템을 쓰면 변경 사항을 검토할 수 있고, 어떤 변경이 버그를 도입했을 때 작동하던 버전의 코드로 되돌아가기도 쉽습니다.

이 프로젝트와 이후 프로젝트를 진행하다 보면 이해하기 어려운 버그를 만나게 될 것이라고 예상해야 합니다. 그런 상황이 오면 유용한 디버깅 팁으로 가득한 디버깅 도구 부록을 다시 읽어 보세요. 다시 속도를 올리는 데 도움이 될 것입니다([디버깅 도구](../06_appendix/06_debugging_tools.md) 참고). 특히 [백트레이스](../06_appendix/06_debugging_tools.md#backtraces) 절은 꼭 읽어보세요. 모든 커널 패닉이나 assertion failure에서 더 많은 정보를 끌어내는 데 도움이 됩니다.
