# 01_introduction

# 프로젝트 3: 가상 메모리

이제 여러분은 Pintos의 내부 동작에 어느 정도 익숙해졌을 것입니다. 여러분의
OS는 적절한 동기화와 함께 여러 실행 스레드를 올바르게 처리할 수 있고, 여러
사용자 프로그램을 동시에 로드할 수 있습니다. 하지만 실행할 수 있는 프로그램의
개수와 크기는 기계의 주 메모리 크기에 의해 제한됩니다. 이번 과제에서는 무한한
메모리가 있는 것 같은 환상을 만들어 이 제한을 없앨 것입니다.

이 과제는 이전 과제 위에 이어서 구현하게 됩니다. project 2의 테스트
프로그램도 project 3에서 동작해야 합니다. 따라서 project 3 작업을 시작하기
전에 project 2 제출물에 있는 버그를 반드시 수정해 두어야 합니다. 그렇지
않으면 그 버그가 project 3에서도 같은 문제를 일으킬 가능성이 매우 큽니다.

project 3는 편의를 위해 단계별로 안내합니다.

## 배경

### 소스 파일

이 프로젝트에서는 `vm` 디렉터리에서 작업합니다. `Makefile`은 `-DVM` 설정을
활성화하도록 업데이트되어 있습니다. 템플릿 코드도 매우 많이 제공됩니다.
***반드시 제공된 템플릿을 따라야 합니다. 즉, 제공된 템플릿을 기반으로 하지***​***않은 코드를 제출하면 0점을 받습니다.***  또한 템플릿에 "DO NOT CHANGE"라고
표시된 부분은 절대 수정하면 안 됩니다. 아래에는 여러분이 수정하게 될 각
템플릿 파일에 대한 설명을 제공합니다.

- `include/vm/vm.h`, `vm/vm.c`

  가상 메모리에 대한 일반 인터페이스를 제공합니다. 헤더 파일에는 여러분의
  가상 메모리 시스템이 지원해야 하는 서로 다른 vm_type -- `VM_UNINIT`,
  `VM_ANON`, `VM_FILE`, `VM_PAGE_CACHE` -- 의 정의와 설명이 들어 있습니다
  (`VM_PAGE_CACHE`는 지금은 무시하세요. 이는 project 4용입니다). 또한 여기서
  보조 페이지 테이블을 구현하게 됩니다(아래 설명 참고).
- `include/vm/uninit.h`, `vm/uninit.c`

  초기화되지 않은 페이지(vm_type = `VM_UNINIT`)에 대한 연산을 제공합니다.
  현재 설계에서는 모든 페이지가 처음에는 초기화되지 않은 페이지로 만들어지고,
  이후 익명 페이지나 파일 기반 페이지로 변환됩니다.
- `include/vm/anon.h`, `vm/anon.c`

  익명 페이지(vm_type = `VM_ANON`)에 대한 연산을 제공합니다.
- `include/vm/file.h`, `vm/file.c`

  파일 기반 페이지(vm_type = `VM_FILE`)에 대한 연산을 제공합니다.
- `include/vm/inspect.h`, `vm/inspect.c`

  채점을 위한 메모리 검사 연산이 들어 있습니다. 이 파일들은 변경하지 마세요.

이 프로젝트에서 작성하는 코드 대부분은 `vm` 디렉터리의 파일들과 이전
프로젝트에서 소개된 파일들에 들어가게 됩니다. 처음 보는 파일은 아마 다음 몇
개 정도일 것입니다.

- `include/devices/block.h`, `devices/block.c`

  블록 장치에 대한 섹터 단위 읽기/쓰기 접근을 제공합니다. 이 인터페이스를
  사용해 스왑 파티션을 블록 장치처럼 다루게 됩니다.

### 메모리 용어

먼저 메모리와 저장장치 관련 용어를 정리하겠습니다. 이 용어들 중 일부는
project 2에서 이미 익숙할 것입니다([가상 메모리 레이아웃](../03_project2_user_programs/01_introduction.md)
참고). 하지만 상당수는 이번에 새롭게 등장합니다.

#### 페이지

페이지(page), 또는 가상 페이지(virtual page)는 길이가 4,096바이트인 연속된
가상 메모리 영역입니다. 이 4,096바이트를 ***페이지 크기(page size)***  라고
합니다. 페이지는 반드시 ***페이지 정렬(page-aligned)***  되어 있어야 합니다.
즉, 페이지 시작 주소는 페이지 크기로 나누어떨어지는 가상 주소여야 합니다.
따라서 64비트 가상 주소의 마지막 12비트는 ***페이지 오프셋(page offset)*** 
또는 그냥 ***오프셋(offset)***  입니다. 상위 비트들은 곧 설명할 페이지
테이블의 인덱스를 나타내는 데 사용됩니다. 64비트 시스템에서는 4단계 페이지
테이블을 사용하므로 가상 주소는 다음과 같은 형태입니다.

```
63          48 47            39 38            30 29            21 20         12 11         0
+-------------+----------------+----------------+----------------+-------------+------------+
| Sign Extend |    Page-Map    | Page-Directory | Page-directory |  Page-Table |    Page    |
|             | Level-4 Offset |    Pointer     |     Offset     |   Offset    |   Offset   |
+-------------+----------------+----------------+----------------+-------------+------------+
              |                |                |                |             |            |
              +------- 9 ------+------- 9 ------+------- 9 ------+----- 9 -----+---- 12 ----+
                                          Virtual Address
```

각 프로세스는 독립적인 사용자(가상) 페이지 집합을 가집니다. 이는 가상 주소
`KERN_BASE` (`0x8004000000`) 아래에 있는 페이지들입니다. 반면
***커널(가상) 페이지(kernel virtual pages)***  집합은 전역적이므로, 어떤
스레드나 프로세스가 실행 중이든 항상 같은 위치에 유지됩니다. 커널은 사용자
페이지와 커널 페이지 모두에 접근할 수 있지만, 사용자 프로세스는 자신의
사용자 페이지에만 접근할 수 있습니다. 자세한 내용은
[가상 메모리 레이아웃](../03_project2_user_programs/01_introduction.md)을
참고하세요.

Pintos는 가상 주소를 다룰 때 유용한 여러 함수를 제공합니다. 자세한 내용은
[가상 주소](../06_appendix/04_virtual_address.md) 절을 참고하세요.

#### 프레임

***프레임(frame)*** , 또는 ***물리 프레임(physical frame)*** , ***페이지***​***프레임(page frame)***  은 연속된 물리 메모리 영역입니다. 페이지와 마찬가지로
프레임도 페이지 크기여야 하고 페이지 정렬되어 있어야 합니다. 따라서 64비트
물리 주소 역시 다음과 같이 ***프레임 번호(frame number)***  와 프레임
***오프셋(offset)***  으로 나눌 수 있습니다.

```
                      12 11         0
+-----------------------+-----------+
|      Frame Number     |   Offset  |
+-----------------------+-----------+
          Physical Address
```

x86-64는 물리 주소에 있는 메모리에 직접 접근하는 방법을 제공하지 않습니다.
Pintos는 커널 가상 메모리를 물리 메모리에 직접 매핑함으로써 이 문제를
우회합니다. 즉, 커널 가상 메모리의 첫 번째 페이지는 물리 메모리의 첫 번째
프레임에, 두 번째 페이지는 두 번째 프레임에, 이런 식으로 매핑됩니다. 따라서
프레임은 커널 가상 메모리를 통해 접근할 수 있습니다.

Pintos는 물리 주소와 커널 가상 주소 사이를 변환하는 함수도 제공합니다. 자세한
내용은 [가상 주소](../06_appendix/04_virtual_address.md)를 참고하세요.

#### 페이지 테이블

***페이지 테이블(page table)***  은 CPU가 가상 주소를 물리 주소로, 즉 페이지를
프레임으로 변환할 때 사용하는 자료구조입니다. 페이지 테이블의 형식은 x86-64
아키텍처에 의해 결정됩니다. Pintos는 `threads/mmu.c`에 페이지 테이블 관리
코드를 제공합니다.

아래 그림은 페이지와 프레임의 관계를 보여 줍니다. 왼쪽의 가상 주소는 페이지
번호와 오프셋으로 이루어져 있습니다. 페이지 테이블은 페이지 번호를 프레임
번호로 변환하고, 여기에 변경되지 않은 오프셋을 결합해 오른쪽의 물리 주소를
얻습니다.

```
                      +----------+
     .--------------->|Page Table|-----------.
    /                 +----------+            |
    |   12 11 0                               V  12 11 0
+---------+----+                         +---------+----+
| Page Nr | Ofs|                         |Frame Nr | Ofs|
+---------+----+                         +---------+----+
 Virt Addr   |                            Phys Addr    ^
              \_______________________________________/
```

#### 스왑 슬롯

***스왑 슬롯(swap slot)***  은 스왑 파티션 안의 페이지 크기 디스크 공간입니다.
프레임과 달리 슬롯의 배치에는 하드웨어 제약이 덜하지만, 그렇게 하는 데
단점이 없으므로 스왑 슬롯도 페이지 정렬하는 것이 좋습니다.

반드시 완전히 분리된 세 자료구조를 구현할 필요는 없습니다. 관련된 자원들을
하나의 통합 자료구조로 전체 또는 일부를 합쳐 구현하는 편이 더 편리할 수도
있습니다.

각 자료구조에 대해서는 각 원소가 어떤 정보를 담아야 하는지 결정해야 합니다.
또한 자료구조의 범위를 로컬(프로세스별)로 둘지, 전역(시스템 전체)으로 둘지,
그 범위 안에서 인스턴스가 몇 개 필요한지도 결정해야 합니다.

설계를 단순하게 하기 위해, 이 자료구조들은 페이지 아웃되지 않는 메모리
(예: `calloc`이나 `malloc`로 할당한 메모리)에 저장해도 됩니다. 이렇게 하면
자료구조들 사이의 포인터가 항상 유효하다는 점을 확신할 수 있습니다.

#### 구현 선택지 (성능 관점)

가능한 구현 방식으로는 배열, 리스트, 비트맵, 해시 테이블 등이 있습니다.
배열은 대체로 가장 단순하지만, 성기게 채워진 배열은 메모리를 낭비합니다.
리스트도 단순하지만, 특정 위치를 찾기 위해 긴 리스트를 순회하면 시간이
낭비됩니다. 배열과 리스트 모두 크기를 조절할 수 있지만, 리스트는 중간 위치의
삽입과 삭제를 더 효율적으로 지원합니다.

Pintos는 `lib/kernel/bitmap.c`와 `include/lib/kernel/bitmap.h`에 비트맵
자료구조를 포함하고 있습니다. 비트맵은 비트 배열이며, 각 비트는 참 또는 거짓
값을 가집니다. 비트맵은 보통 동일한 형태의 자원 집합의 사용 여부를 추적하는
데 쓰입니다. 자원 n이 사용 중이면 비트맵의 ***n*** 번째 비트가 참입니다.
Pintos의 비트맵은 크기가 고정되어 있지만, 필요하다면 크기 변경을 지원하도록
직접 확장할 수 있습니다.

Pintos는 해시 테이블 자료구조도 포함하고 있습니다([해시 테이블](../06_appendix/08_hash_table.md)
참고). Pintos의 해시 테이블은 다양한 테이블 크기 범위에서 삽입과 삭제를
효율적으로 지원합니다.

더 복잡한 자료구조가 더 나은 성능이나 다른 이점을 줄 수도 있지만, 구현을
불필요하게 복잡하게 만들 가능성도 있습니다. 따라서 설계의 일부로 고급
자료구조(예: 균형 이진 트리)를 구현하는 것은 권장하지 않습니다.

### 보조 페이지 테이블 관리

***보조(supplemental)***  페이지 테이블은 각 페이지에 대한 추가 정보를 담아
페이지 테이블을 보완합니다. 페이지 테이블 형식 자체의 한계 때문에 이
자료구조가 필요합니다. 이런 자료구조 역시 흔히 "page table"이라고 불리기
때문에, 혼동을 줄이기 위해 여기서는 "supplemental"이라는 단어를 붙입니다.

보조 페이지 테이블은 최소 두 가지 목적으로 사용됩니다. 가장 중요한 용도는
페이지 폴트가 발생했을 때, 커널이 보조 페이지 테이블에서 문제가 된 가상
페이지를 찾아 그 자리에 어떤 데이터가 있어야 하는지 알아내는 것입니다.
두 번째는 프로세스가 종료될 때 커널이 어떤 자원을 해제해야 하는지 판단하기
위해 보조 페이지 테이블을 참고하는 것입니다.

#### 보조 페이지 테이블의 구성

보조 페이지 테이블은 원하는 방식으로 구성해도 됩니다. 구성 방식에는 적어도 두
가지 기본 접근이 있습니다. 세그먼트 단위로 구성하거나, 페이지 단위로 구성하는
방식입니다. 여기서 세그먼트는 연속된 페이지들의 묶음, 즉 실행 파일이나
메모리 매핑된 파일이 들어 있는 메모리 영역을 뜻합니다.

선택 사항으로, 보조 페이지 테이블의 구성원을 추적하는 데 페이지 테이블 자체를
사용할 수도 있습니다. 이 방식을 사용하려면 `threads/mmu.c`의 Pintos 페이지
테이블 구현을 수정해야 합니다. 이 방법은 숙련된 학생에게만 권장합니다.

#### 페이지 폴트 처리

보조 페이지 테이블을 사용하는 가장 중요한 곳은 페이지 폴트 핸들러입니다.
project 2에서는 페이지 폴트가 발생하면 언제나 커널 또는 사용자 프로그램의
버그를 뜻했습니다. 하지만 project 3에서는 더 이상 그렇지 않습니다. 이제
페이지 폴트는 파일이나 스왑 슬롯으로부터 페이지를 가져와야 한다는 신호일 수도
있습니다. 이런 경우를 처리하려면 더 정교한 페이지 폴트 핸들러를 구현해야
합니다. `userprog/exception.c`의 `page_fault()`가 여러분이 구현할 페이지
폴트 핸들러인 `vm/vm.c`의 `vm_try_handle_fault()`를 호출합니다. 여러분의
페이지 폴트 핸들러는 대략 다음 순서로 동작해야 합니다.

1. 보조 페이지 테이블에서 폴트가 난 페이지를 찾습니다. 메모리 참조가
   유효하다면, 보조 페이지 테이블 엔트리를 사용해 그 페이지에 들어가야 할
   데이터를 찾습니다. 이 데이터는 파일 시스템에 있을 수도 있고, 스왑 슬롯에
   있을 수도 있으며, 단순히 0으로만 채워진 페이지일 수도 있습니다. 공유
   기능(즉, Copy-on-Write)을 구현했다면, 페이지 데이터가 이미 어떤 프레임에
   들어 있지만 페이지 테이블에는 아직 없을 수도 있습니다. 보조 페이지
   테이블이 해당 주소에 사용자 프로세스가 기대할 만한 데이터가 없다고
   나타내거나, 페이지가 커널 가상 메모리 범위에 있거나, 읽기 전용 페이지에
   쓰기를 시도한 경우라면 그 접근은 잘못된 것입니다. 잘못된 접근이 발생하면
   프로세스를 종료하고, 그에 따라 해당 프로세스의 모든 자원도 해제해야
   합니다.
2. 페이지를 저장할 프레임을 얻습니다. 공유를 구현했다면 필요한 데이터가 이미
   프레임에 있을 수 있으므로, 그 프레임을 찾아낼 수 있어야 합니다.
3. 파일 시스템이나 스왑에서 읽어 오거나, 0으로 채우는 등의 방식으로 데이터를
   프레임에 가져옵니다. 공유를 구현했다면 필요한 페이지가 이미 프레임에
   있을 수 있으므로 이 단계에서는 아무 작업도 필요 없을 수 있습니다.
4. 폴트가 발생한 가상 주소에 대한 페이지 테이블 엔트리가 해당 물리 페이지를
   가리키도록 설정합니다. `threads/mmu.c`의 함수들을 사용할 수 있습니다.

### 프레임 테이블 관리

프레임 테이블은 각 프레임마다 하나의 엔트리를 가집니다. 프레임 테이블의 각
엔트리에는 현재 그 프레임을 차지하고 있는 페이지가 있다면 그 페이지에 대한
포인터와, 여러분이 선택한 다른 데이터가 들어갑니다. 프레임 테이블은 빈
프레임이 없을 때 어떤 페이지를 축출할지 결정하는 방식으로 Pintos가 축출
정책을 효율적으로 구현할 수 있게 해 줍니다.

사용자 페이지에 쓰이는 프레임은 반드시 "user pool"에서
`palloc_get_page(PAL_USER)`를 호출해 얻어야 합니다. `PAL_USER`를 사용해야
"kernel pool"에서 메모리를 할당하는 일을 피할 수 있으며, 그렇지 않으면 일부
테스트 케이스가 예상치 못하게 실패할 수 있습니다. 프레임 테이블 구현의 일부로
`palloc.c`를 수정한다면, 두 풀의 구분은 반드시 유지해야 합니다.

프레임 테이블에서 가장 중요한 연산은 사용되지 않는 프레임을 얻는 것입니다.
프레임이 비어 있다면 이것은 쉽습니다. 하지만 비어 있는 프레임이 없다면, 어떤
페이지를 그 프레임에서 축출하여 프레임을 비워야 합니다.

스왑 슬롯을 새로 할당하지 않고는 어떤 프레임도 축출할 수 없는데 스왑도 가득
찬 상태라면, 커널을 panic 시키세요. 실제 OS는 이런 상황을 예방하거나
복구하기 위해 매우 다양한 정책을 사용하지만, 그런 정책은 이 프로젝트의 범위를
벗어납니다.

축출 과정은 대략 다음 단계로 이루어집니다.

1. 페이지 교체 알고리즘을 사용해 축출할 프레임을 하나 고릅니다. 아래에서
   설명하는 페이지 테이블의 "accessed" 비트와 "dirty" 비트가 유용하게 쓰일
   것입니다.
2. 그 프레임을 참조하는 모든 페이지 테이블에서 해당 프레임에 대한 참조를
   제거합니다. 공유를 구현하지 않았다면, 일반적으로 한 시점에 하나의 페이지만
   하나의 프레임을 참조해야 합니다.
3. 필요하다면 페이지를 파일 시스템이나 스왑에 기록합니다. 그 후 축출된
   프레임은 다른 페이지를 저장하는 데 사용할 수 있습니다.

#### Accessed 비트와 Dirty 비트

x86-64 하드웨어는 각 페이지의 페이지 테이블 엔트리(PTE)에 있는 한 쌍의 비트를
통해 페이지 교체 알고리즘 구현을 도와줍니다. 어떤 페이지에 대해 읽기 또는
쓰기가 발생하면 CPU는 그 페이지의 PTE에 있는 accessed 비트를 1로 설정하고,
쓰기 발생 시에는 ***dirty 비트*** 도 1로 설정합니다. CPU는 이 비트들을 다시
0으로 되돌리지는 않지만, OS는 그렇게 할 수 있습니다.

여기서 주의해야 할 것이 `aliases`입니다. 즉, 같은 프레임을 가리키는 둘
이상의 페이지를 뜻합니다. alias가 있는 프레임이 접근되면, accessed 비트와
dirty 비트는 한 페이지 테이블 엔트리(실제로 접근에 사용된 페이지의 엔트리)에만
업데이트됩니다. 나머지 alias들의 accessed 비트와 dirty 비트는 갱신되지
않습니다.

Pintos에서는 모든 사용자 가상 페이지가 그에 대응하는 커널 가상 페이지와
alias 관계를 가집니다. 이 alias들을 어떻게든 관리해야 합니다. 예를 들어,
양쪽 주소 모두에 대해 accessed 비트와 dirty 비트를 확인하고 업데이트하는
방법이 있습니다. 또는 커널이 사용자 데이터를 오직 사용자 가상 주소를 통해서만
접근하도록 하여 이 문제를 피할 수도 있습니다.

공유를 구현했거나 여러분의 코드에 버그가 있는 경우가 아니라면, 다른 alias는
생기지 않아야 합니다.

accessed 비트와 dirty 비트를 다루는 함수에 대한 자세한 내용은
[페이지 테이블의 Accessed/Dirty 비트](../06_appendix/05_page_table.md) 절을
참고하세요.

### 스왑 테이블 관리

스왑 테이블은 사용 중인 스왑 슬롯과 비어 있는 스왑 슬롯을 추적합니다. 프레임에
있던 페이지를 스왑 파티션으로 축출할 때 사용하지 않은 스왑 슬롯을 선택할 수
있어야 합니다. 또한 페이지를 다시 읽어 들였을 때나, 해당 페이지를 스왑해 둔
프로세스가 종료되었을 때 스왑 슬롯을 해제할 수 있어야 합니다.

`vm/build` 디렉터리에서 `pintos-mkdisk swap.dsk --swap-size=n` 명령을 사용하면
`n`MB 크기의 스왑 파티션을 포함한 `swap.dsk` 디스크를 만들 수 있습니다. 그
뒤에는 pintos를 실행할 때 `swap.dsk`가 자동으로 추가 디스크로 연결됩니다.
또는 `--swap-size=n` 옵션을 사용해 한 번의 실행에 한해 임시 `n`MB 스왑
디스크를 사용하도록 할 수도 있습니다.

스왑 슬롯은 지연 할당되어야 합니다. 즉, 실제로 축출이 필요할 때만 할당해야
합니다. 프로세스 시작 시점에 실행 파일의 데이터 페이지를 읽어 와서 곧바로
스왑에 써 두는 방식은 lazy하지 않습니다. 특정 페이지를 저장하기 위해 스왑
슬롯을 미리 예약해서도 안 됩니다.

스왑 슬롯의 내용이 프레임으로 다시 읽혀 오면 그 슬롯을 해제하세요.

### 메모리 매핑 파일 관리

파일 시스템에는 가장 흔하게는 `read`와 `write` 시스템 콜을 통해 접근합니다.
그 외의 보조 인터페이스로, `mmap` 시스템 콜을 사용해 파일을 가상 페이지에
"매핑"할 수 있습니다. 그러면 프로그램은 파일 데이터를 메모리 명령으로 직접
다룰 수 있습니다. 예를 들어 `foo` 파일의 크기가 `0x1000`바이트(4 kB, 즉 한
페이지)라고 가정합시다. `foo`를 주소 `0x5000`부터 메모리에 매핑하면,
`0x5000. . .0x5fff` 범위에 대한 메모리 접근은 `foo`의 대응하는 바이트들에
접근하게 됩니다.

다음은 `mmap`을 사용해 파일을 콘솔에 출력하는 프로그램입니다. 커맨드라인에서
지정한 파일을 열고, 이를 가상 주소 `0x10000000`에 매핑한 뒤, 매핑된 데이터를
콘솔(fd 1)에 기록하고, 마지막으로 파일 매핑을 해제합니다.

```c
#include <stdio.h>
#include <syscall.h>
int main (int argc UNUSED, char *argv[])
{
  void *data = (void *) 0x10000000;                 /* Address at which to map. */
  int fd = open (argv[1]);                          /* Open file. */
  void *map = mmap (data, filesize (fd), 0, fd, 0); /* Map file. */
  write (1, data, filesize (fd));                   /* Write file to console. */
  munmap (map);                                     /* Unmap file (optional). */
  return 0;
}
```

제출한 구현은 메모리 매핑 파일이 사용하는 메모리를 추적할 수 있어야 합니다.
이것은 매핑된 영역에서 페이지 폴트를 올바르게 처리하고, 매핑된 파일이 프로세스
안의 다른 세그먼트와 겹치지 않도록 보장하는 데 필요합니다.
