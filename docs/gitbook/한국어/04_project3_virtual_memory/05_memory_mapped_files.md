# 05_memory_mapped_files

### 메모리 매핑 파일

이 절에서는 메모리 매핑 페이지를 구현합니다. 익명 페이지와 달리, 메모리 매핑
페이지는 파일 기반 매핑입니다. 페이지의 내용은 기존 파일의 데이터를
반영합니다. 페이지 폴트가 발생하면 물리 프레임이 즉시 할당되고, 파일에서
메모리로 내용이 복사됩니다. 메모리 매핑 페이지가 unmap되거나 swap out될 때는,
그 내용에 대한 모든 변경 사항이 파일에 반영됩니다.

#### `mmap` 및 `munmap` 시스템 콜

**메모리 매핑 파일을 위한 두 시스템 콜인 **​**​`mmap`​**​**과 **​**​`munmap`​**​**을 구현하세요.** 
여러분의 VM 시스템은 mmap 영역의 페이지를 lazy하게 로드해야 하며, 매핑의
backing store로 mmap된 파일 자체를 사용해야 합니다. 이 두 시스템 콜을
구현하기 위해 `vm/file.c`에 정의된 `do_mmap`과 `do_munmap`을 구현하고
사용해야 합니다.

---

```c
void *mmap (void *addr, size_t length, int writable, int fd, off_t offset);
```

> `fd`로 열린 파일의 `offset` 바이트부터 시작하는 `length`바이트를 프로세스의
> 가상 주소 공간 안 `addr`에 매핑합니다. 파일 전체는 `addr`에서 시작하는
> 연속된 가상 페이지들에 매핑됩니다. 파일 길이가 PGSIZE의 배수가 아니라면,
> 마지막으로 매핑된 페이지의 일부 바이트는 파일 끝을 넘어 "튀어나오게"
> 됩니다. 이런 바이트들은 페이지가 fault in될 때 0으로 채우고, 디스크에 다시
> 쓸 때는 버리세요. 성공하면 이 함수는 파일이 매핑된 가상 주소를 반환합니다.
> 실패하면 파일 매핑에 유효하지 않은 주소인 NULL을 반드시 반환해야 합니다.

`mmap` 호출은 `fd`로 열린 파일의 길이가 0바이트인 경우 실패할 수 있습니다.
또한 `addr`이 페이지 정렬되어 있지 않거나, 매핑하려는 페이지 범위가 기존에
매핑된 어떤 페이지 집합과라도 겹치면 반드시 실패해야 합니다. 여기에는 스택과
실행 파일 로드 시 매핑된 페이지도 포함됩니다. Linux에서는 `addr`이 NULL이면
커널이 적절한 매핑 주소를 찾습니다. 하지만 단순화를 위해 여기서는 주어진
`addr`에 그대로 `mmap`을 시도하면 됩니다. 따라서 `addr`이 0이면 반드시
실패해야 합니다. 일부 Pintos 코드가 가상 페이지 0이 매핑되지 않았다고
가정하기 때문입니다. `length`가 0인 경우에도 `mmap`은 실패해야 합니다.
마지막으로 콘솔 입력과 출력을 나타내는 파일 디스크립터는 매핑할 수 없습니다.

메모리 매핑 페이지 역시 익명 페이지와 마찬가지로 lazy하게 할당되어야 합니다.
`vm_alloc_page_with_initializer`나 `vm_alloc_page`를 사용해 페이지 객체를
만들 수 있습니다.

---

```c
void munmap (void *addr);
```

> 지정한 주소 범위 `addr`에 대한 매핑을 해제합니다. `addr`은 같은 프로세스가
> 이전에 `mmap`으로 얻었고 아직 해제하지 않은 가상 주소여야 합니다.

프로세스가 `exit`로 종료하든 다른 방식으로 종료하든, 모든 매핑은 프로세스
종료 시 암묵적으로 해제됩니다. 매핑이 해제될 때, 명시적이든 암묵적이든,
프로세스가 쓴 페이지는 모두 파일에 다시 기록되어야 하며, 쓰지 않은 페이지는
기록되면 안 됩니다. 그런 다음 이 페이지들은 프로세스의 가상 페이지 목록에서
제거됩니다.

파일을 닫거나 제거해도 그 파일의 어떤 매핑도 해제되지 않습니다. Unix 관례에
따라, 한 번 만들어진 매핑은 `munmap`이 호출되거나 프로세스가 종료할 때까지
유효합니다. 자세한 내용은
[열려 있는 파일 제거](../03_project2_user_programs/08_faq.md)를 참고하세요.
각 매핑마다 파일에 대한 별도이며 독립적인 참조를 얻기 위해 `file_reopen`
함수를 사용해야 합니다.

둘 이상의 프로세스가 같은 파일을 매핑해도, 그들이 일관된 데이터를 보게 만들
필요는 없습니다. Unix는 두 매핑이 같은 물리 페이지를 공유하게 하여 이 문제를
다루며, `mmap` 시스템 콜에는 페이지를 공유할지 private하게 둘지(즉
copy-on-write) 클라이언트가 지정할 수 있는 인자도 있습니다.

필요에 따라 `vm/vm.c`의 `vm_file_init`와 `vm_file_initializer`를 수정해도
됩니다.

---

```c
void vm_file_init (void);
```

> 파일 기반 페이지 서브시스템을 초기화합니다. 이 함수에서는 파일 기반
> 페이지와 관련된 어떤 설정이든 할 수 있습니다.

---

```c
bool file_backed_initializer (struct page *page, enum vm_type type, void *kva);
```

> 파일 기반 페이지를 초기화합니다. 이 함수는 먼저 파일 기반 페이지를 위한
> 핸들러를 `page->operations`에 설정합니다. 메모리를 뒷받침하는 파일처럼,
> page struct의 일부 정보를 갱신해야 할 수도 있습니다.

---

```c
static void file_backed_destroy (struct page *page);
```

> 연관된 파일을 닫아 파일 기반 페이지를 파괴합니다. 내용이 dirty하다면,
> 변경 사항을 반드시 파일에 다시 써야 합니다. 이 함수에서 page struct
> 자체를 해제할 필요는 없습니다. `file_backed_destroy`를 호출한 쪽에서
> 처리해야 합니다.
