# 06_debugging_tools

# 디버깅 도구

Pintos를 디버깅할 때 사용할 수 있는 도구는 많습니다. 이 부록에서는 그중 몇 가지를 소개합니다.

## `printf()`

`printf()`의 가치를 과소평가하지 마십시오. Pintos에서 `printf()`는 구현 방식 덕분에, 커널 스레드에서든 인터럽트 핸들러에서든, 그리고 어떤 락을 잡고 있느냐와 거의 무관하게 커널의 사실상 어디서나 호출할 수 있습니다. `printf()`는 단순히 데이터를 살펴보는 데만 유용한 것이 아닙니다. 커널이 유용한 에러 메시지 없이 크래시가 나거나 panic하는 경우에도, 언제 어디서 문제가 발생하는지 알아내는 데 도움이 됩니다. 전략은 문제가 있다고 의심되는 코드 곳곳에 서로 다른 문자열(예: "<1>", "<2>", . . . )을 넣은 `printf()` 호출을 흩뿌리는 것입니다. `<1>`조차 출력되지 않으면 그 지점보다 앞에서 문제가 생긴 것이고, `<1>`은 보이지만 `<2>`가 보이지 않으면 그 두 지점 사이에서 문제가 생긴 것입니다. 이런 식으로 파악한 내용을 바탕으로, 새롭게 좁혀진 더 작은 의심 구간에 `printf()` 호출을 더 넣을 수 있습니다. 결국 문제를 한 문장까지 좁힐 수 있습니다. 관련 기법으로 [Triple Faults](06_debugging_tools.md#triple-faults)를 참고하십시오.

## ASSERT

assertion은 원래라면 더 늦게 발견됐을 문제를 초기에 잡아낼 수 있기 때문에 유용합니다. 이상적으로는 각 함수가 인자의 유효성을 확인하는 assertion 집합으로 시작해야 합니다. (함수의 지역 변수 초기화식은 assertion을 검사하기 전에 평가되므로, 초기화식 안에서는 인자가 유효하다고 가정하지 않도록 주의해야 합니다.) 문제가 생길 가능성이 높다고 의심되는 지점에는 함수 본문 곳곳에도 assertion을 넣을 수 있습니다. 특히 루프 불변식을 검사할 때 유용합니다.

Pintos는 assertion 검사용으로 `<debug.h>`에 정의된 `ASSERT` 매크로를 제공합니다.

```c
#define ASSERT(expression) { /* Omit details */ }
```

`expression`의 값을 검사합니다. 이 값이 0(false)으로 평가되면 커널이 panic합니다. panic 메시지에는 실패한 `expression`, 해당 파일과 줄 번호, 그리고 backtrace가 포함되며, 이는 문제를 찾는 데 도움이 됩니다. 자세한 내용은 [백트레이스](#backtraces)를 참고하십시오.

## 함수 및 매개변수 속성

`<debug.h>`에 정의된 이 매크로들은 함수나 함수 매개변수의 특별한 속성을 컴파일러에 알려 줍니다. 이 매크로의 확장은 GCC 전용입니다.

---

```c
#define UNUSED { /* Omit details */ }
```

> 함수 매개변수 뒤에 붙여, 해당 매개변수가 함수 내부에서 사용되지 않을 수도 있음을 컴파일러에 알려 줍니다. 그렇지 않으면 발생할 경고를 억제합니다.

---

```c
#define NO_RETURN { /* Omit details */ }
```

> 함수 프로토타입 뒤에 붙여, 해당 함수가 절대 반환하지 않음을 컴파일러에 알려 줍니다. 이를 통해 컴파일러는 경고와 코드 생성을 더 정교하게 조정할 수 있습니다.

---

```c
#define NO_INLINE { /* Omit details */ }
```

> 함수 프로토타입 뒤에 붙여, 컴파일러가 해당 함수를 절대로 인라인으로 넣지 않도록 합니다. 가끔 backtrace의 품질을 높이는 데 유용합니다(아래 참고).

---

```c
#define PRINTF_FORMAT(format, first) { /* Omit details */ }
```

> 함수 프로토타입 뒤에 붙여, 해당 함수가 `printf()` 스타일의 format 문자열을 format번째 인자(1부터 시작)로 받고, 그에 대응하는 값 인자들은 first번째 인자부터 시작한다는 사실을 컴파일러에 알려 줍니다. 이렇게 하면 잘못된 인자 타입을 넘겼을 때 컴파일러가 알려 줄 수 있습니다.

## Backtraces

커널이 panic하면 "backtrace"를 출력합니다. 즉, panic 당시 실행 중이던 함수 내부 주소 목록을 통해 프로그램이 현재 위치까지 어떤 경로로 왔는지 요약해서 보여 줍니다. 코드의 어느 지점에서든 `<debug.h>`에 선언된 `debug_backtrace()`를 호출해 backtrace를 출력할 수도 있습니다. 마찬가지로 `<debug.h>`에 선언된 `debug_backtrace_all()`은 모든 스레드의 backtrace를 출력합니다.

backtrace 안의 주소는 원시 16진수 숫자로 나열되므로 해석하기 어렵습니다. 이를 함수 이름과 소스 파일 줄 번호로 바꿔 주는 `backtrace`라는 도구를 제공합니다. 첫 번째 인자로는 `kernel.o`의 이름을 주고, 나머지 인자로는 backtrace를 구성하는 16진수 숫자들(`0x` 접두사 포함)을 넘기면 됩니다. 그러면 각 주소에 대응하는 함수 이름과 소스 파일 줄 번호를 출력합니다.

변환된 backtrace가 깨져 보이거나, 함수 A 위에 함수 B가 있는데 B가 A를 호출하지 않는 것처럼 말이 안 된다면, backtrace는 스택에서 추출되므로 커널 스레드의 스택을 손상시키고 있다는 강한 신호입니다. 또는 `backtrace`에 넘긴 `kernel.o`가 실제로 그 backtrace를 만들어 낸 커널과 다른 파일일 수도 있습니다.

스택 손상이 전혀 없어도 backtrace가 혼란스러울 때가 있습니다. 컴파일러 최적화가 예상 밖의 동작을 만들 수 있기 때문입니다. 함수가 다른 함수를 마지막 동작으로 호출한 경우(tail call), 호출한 함수는 backtrace에 아예 나타나지 않을 수 있습니다. 비슷하게 함수 A가 절대 반환하지 않는 다른 함수 B를 호출하면, 컴파일러 최적화 때문에 backtrace에 A 대신 무관한 함수 C가 나타날 수 있습니다. 함수 C는 단지 메모리에서 A 바로 뒤에 우연히 놓여 있던 함수일 뿐입니다. 스레드 프로젝트에서는 테스트 실패 backtrace에서 이런 현상을 자주 볼 수 있습니다.

### 예시

다음은 예시입니다. Pintos가 아래와 같은 호출 스택(call stack)을 출력했다고 가정해 보겠습니다. 이 예시는 파일 시스템 프로젝트의 실제 Pintos 제출물에서 가져온 것입니다:

```bash
Call stack: 0x800426eff 0x8004202fb 0x80042dc22 0x80042cf67 0x800422319
0x80042325a 0x40012c 0x400a96 0x400ac8.
```

그러면 아래와 같이 `backtrace` 유틸리티를 호출하고, backtrace 정보를 명령줄에 잘라 붙여 넣으면 됩니다. 여기서는 `kernel.o`가 현재 디렉터리에 있다고 가정합니다. 물론 아래 내용은 여기서처럼 줄이 넘어가더라도 실제로는 하나의 셸 명령줄에 모두 입력해야 합니다:

```bash
backtrace 0x800426eff 0x8004202fb 0x80042dc22 0x80042cf67 0x800422319
0x80042325a 0x400012c 0x400a96 0x4000ac8
```

그러면 backtrace 출력은 대략 다음과 같을 것입니다:

```bash
0x0000000800426eff: debug_panic (lib/debug.c:86)
0x00000008004202fb: file_seek (filesys/file.c:405)
0x000000080042dc22: seek (userprog/syscall.c:744)
0x000000080042cf67: syscall_handler (userprog/syscall.c:444)
0x0000000800422319: intr_handler (threads/interrupt.c:334)
0x000000080042325a: intr_entry (threads/intr-stubs.S:38)
0x000000000040012c: (unknown)
0x0000000000400a96: (unknown)
0x0000000000400ac8: (unknown)
```

(여러분 자신의 커널 바이너리에서 위 명령을 실행하면, 컴파일한 소스 코드와 사용한 컴파일러가 다를 가능성이 높기 때문에 정확히 같은 주소는 아마 보이지 않을 것입니다.)
backtrace의 첫 번째 줄은 커널 panic을 구현하는 함수 `debug_panic()`를 가리킵니다. backtrace는 보통 커널 panic의 결과로 생기므로, `debug_panic()`은 backtrace에 가장 먼저 나타나는 함수인 경우가 많습니다.
두 번째 줄은 panic을 일으킨 함수가 `file_seek()`였음을 보여 줍니다. 이 경우 원인은 assertion 실패입니다. 이 예시에 사용된 소스 코드 트리에서 `filesys/file.c`의 405번째 줄은 다음 assertion입니다.

```
ASSERT (file_ofs >= 0);
```

(이 줄은 assertion failure 메시지에서도 함께 언급되었습니다.) 따라서 `file_seek()`은 음수 파일 오프셋 인자를 넘겨받았기 때문에 panic했습니다.
세 번째 줄은 `seek()`가 오프셋 인자를 검증하지 않은 채 `file_seek()`를 호출했음을 보여 줍니다. 이 제출물에서 `seek()`는 seek 시스템 콜을 구현합니다. 네 번째 줄은 시스템 콜 핸들러인 `syscall_handler()`가 `seek()`를 호출했음을 보여 줍니다. 다섯 번째와 여섯 번째 줄은 인터럽트 핸들러 진입 경로입니다. 나머지 줄은 `KERN_BASE` 아래 주소들에 해당합니다. 즉, 이 주소들은 커널이 아니라 사용자 프로그램 안의 주소를 가리킵니다.

## GDB

GDB 디버거의 감독 아래에서 Pintos를 실행할 수 있습니다. 먼저 `--gdb` 옵션으로 Pintos를 시작합니다. 예를 들어 `pintos --gdb -- run mytest`와 같습니다. 그런 다음 같은 머신에서 두 번째 터미널을 열고, gdb를 사용해 `kernel.o`에 대해 GDB를 실행합니다.

```
gdb kernel.o
```

그리고 다음 GDB 명령을 입력합니다:

```
target remote localhost:1234
```

이제 GDB는 로컬 네트워크 연결을 통해 시뮬레이터와 연결됩니다. 이제 일반적인 GDB 명령을 무엇이든 사용할 수 있습니다. `c` 명령을 내리면 시뮬레이션된 BIOS가 제어권을 가져가 Pintos를 적재한 뒤, Pintos가 평소와 같은 방식으로 실행됩니다. 어느 지점에서든 Ctrl+C로 과정을 멈출 수 있습니다.

### GDB 사용하기

터미널 명령 프롬프트에서 `info gdb`를 입력하면 GDB 매뉴얼을 읽을 수 있습니다. 아래는 자주 유용하게 쓰이는 GDB 명령 몇 가지입니다:

- c [GDB Command]

> `Ctrl+C`를 누르거나 다음 중단점(breakpoint)에 도달할 때까지 실행을 계속합니다.

- break *function* [GDB Command]
- break *file:line* [GDB Command]
- break *address* [GDB Command]

> 함수, 파일 안의 줄, 또는 주소에 중단점(breakpoint)을 설정합니다. (16진수 주소를 지정하려면 `0x` 접두사를 사용하십시오.)
> Pintos가 실행을 시작할 때 GDB가 멈추게 하려면 `break main`을 사용하십시오.

- p *expression* [GDB Command]

> 주어진 expression을 평가하고 값을 출력합니다. expression 안에 함수 호출이 들어 있으면 그 함수는 실제로 실행됩니다.

- l *address* [GDB Command]

> 주소 주변의 코드 몇 줄을 보여 줍니다. (16진수 주소를 지정하려면 `0x` 접두사를 사용하십시오.)

- bt [GDB Command]

> 위에서 설명한 backtrace 프로그램 출력과 비슷한 스택 backtrace를 출력합니다.

- p/a *address* [GDB Command]

> 해당 주소를 차지하는 함수나 변수의 이름을 출력합니다. (16진수 주소를 지정하려면 `0x` 접두사를 사용하십시오.)

- diassemble *function* [GDB Command]

> 함수를 디스어셈블합니다.

여기까지 읽은 분들을 위한 추가 팁이 하나 있습니다. [pwndbg](https://github.com/pwndbg/pwndbg) 같은 GDB 스크립트를 설치해 보십시오. 디버깅이 훨씬 편해집니다.

### FAQ

- Emacs 안에서 GDB를 사용할 수 있나요?

> 네, 가능합니다. Emacs는 GDB를 서브프로세스로 실행하는 특별한 지원을 제공합니다.
> `M-x gdb`를 입력한 다음 프롬프트에서 여러분의 pintos-gdb 명령을 입력하십시오. Emacs 매뉴얼에는 "Debuggers"라는 제목의 섹션에서 이러한 디버깅 기능 사용법이 설명되어 있습니다.

- GDB가 이상하게 동작합니다.

> GDB를 사용하는 동안 이상한 동작이 보인다면 가능성은 세 가지입니다:
> 여러분이 수정한 Pintos의 버그, GDB에 대한 Bochs 인터페이스나 GDB 자체의 버그, 또는 원래 Pintos 코드의 버그입니다. 첫 번째와 두 번째는 상당히 그럴 가능성이 높으므로 둘 다 진지하게 고려해야 합니다. 세 번째는 그보다 가능성이 낮기를 바라지만, 역시 가능성은 있습니다.

## Triple Faults

페이지 폴트 핸들러처럼 CPU 예외 핸들러가 없거나 고장 나서 호출할 수 없으면, CPU는 "double fault" 핸들러를 호출하려고 시도합니다. 그런데 double fault 핸들러 역시 없거나 고장 나 있다면 이를 "triple fault"라고 합니다. triple fault가 발생하면 CPU는 즉시 리셋됩니다.

따라서 머신이 루프를 돌며 재부팅되는 상황에 빠졌다면, 아마 "triple fault"일 가능성이 큽니다. triple fault 상황에서는 `printf()`를 디버깅에 쓰지 못할 수도 있습니다. `printf()`에 필요한 모든 것이 초기화되기도 전에 재부팅이 일어날 수 있기 때문입니다.

triple fault를 디버깅하는 방법도 있습니다.

Pintos 코드 어딘가를 골라 그 자리에 무한 루프 `for (;;);`를 넣고, 다시 컴파일한 다음 실행해 보십시오. 가능성은 크게 두 가지입니다:

- 머신이 재부팅되지 않고 멈춥니다. 이런 경우 무한 루프가 실제로 실행되었다는 뜻입니다. 즉, 재부팅을 일으킨 원인은 여러분이 무한 루프를 넣은 지점 *이후* 에 있어야 합니다. 이제 코드 흐름상 더 뒤쪽으로 무한 루프를 옮기십시오.
- 머신이 루프를 돌며 재부팅됩니다. 이런 경우 머신이 무한 루프까지 도달하지 못했다는 뜻입니다. 따라서 재부팅을 일으킨 원인은 여러분이 무한 루프를 넣은 지점 *이전* 에 있어야 합니다. 이제 코드 흐름상 더 앞쪽으로 무한 루프를 옮기십시오.

  "이진 탐색(binary search)" 하듯 무한 루프 위치를 옮겨 가면, 이 기법으로 모든 것이 잘못되기 시작하는 정확한 지점을 찾아낼 수 있습니다. 길어야 몇 분이면 충분합니다.

## 팁

`threads/palloc.c`의 페이지 할당기와 `threads/malloc.c`의 블록 할당기는 메모리를 해제할 때 해당 메모리의 모든 바이트를 `'0xcc'`로 채웁니다. 따라서 `0xcccccccc` 같은 포인터를 역참조하려는 시도나 `0xcc`에 대한 다른 참조가 보인다면, 이미 해제된 페이지를 다시 사용하려는 것일 가능성이 큽니다. 또한 바이트 `0xcc`는 CPU에서 "invoke interrupt 3"에 해당하는 연산 코드(opcode)이므로, `Interrupt 0x03 (#BP Breakpoint Exception)` 같은 에러가 보인다면 Pintos가 이미 해제된 페이지나 블록의 코드를 실행하려 했다는 뜻입니다. `sec_no < d->capacity` 식에 대한 assertion 실패는 Pintos가 이미 닫히고 해제된 inode를 통해 파일에 접근하려 했음을 뜻합니다. inode를 해제하면 시작 섹터 번호가 `0xcccccccc`로 지워지는데, 이는 약 1.6 TB보다 작은 디스크에서는 유효한 섹터 번호가 아닙니다.
