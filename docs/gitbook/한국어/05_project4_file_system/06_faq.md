# 06_faq

# FAQ

### BLOCK_SECTOR_SIZE는 바뀔 수 있나요?

아니요. BLOCK_SECTOR_SIZE는 512로 고정되어 있습니다. IDE 디스크에서는 이 값이 하드웨어의 고정 속성입니다. 다른 디스크는 반드시 512바이트 sector를 갖는 것은 아니지만, 단순화를 위해 Pintos는 그런 디스크만 지원합니다.

### 인덱스 기반 파일 FAQ

#### 우리가 지원해야 하는 가장 큰 파일 크기는 얼마인가요?

우리가 만드는 파일 시스템 파티션의 크기는 8 MB 이하입니다. 하지만 메타데이터를 수용할 공간이 필요하므로, 개별 파일의 크기는 파티션 전체보다 작아야 합니다.
inode 구성을 결정할 때 이 점을 고려해야 합니다.

### 하위 디렉터리 FAQ

#### `a//b` 같은 파일 이름은 어떻게 해석해야 하나요?

연속된 여러 슬래시는 하나의 슬래시와 같으므로, 이 파일 이름은 `a/b`와 같습니다.

#### `/../x` 같은 파일 이름은 어떻게 되나요?

루트 디렉터리는 자기 자신을 부모로 가지므로, 이것은 `/x`와 같습니다.

#### `/`로 끝나는 파일 이름은 어떻게 처리해야 하나요?

대부분의 Unix 시스템은 디렉터리 이름 끝의 슬래시는 허용하고, 그 외 이름이 슬래시로 끝나면 거부합니다. 우리도 이 동작을 허용할 것이며, 단순히 슬래시로 끝나는 이름을 거부하는 구현도 허용합니다.

### 버퍼 캐시 FAQ

#### `struct inode` 안에 `struct inode_disk`를 보관해도 되나요?

64-block 제한의 목적은 캐시된 파일 시스템 데이터의 양에 상한을 두는 것입니다. 파일 데이터든 메타데이터든, 디스크 데이터 한 블록을 커널 메모리 어디에든 보관한다면 그 역시 64-block 제한에 포함해야 합니다. `length`나 `sector_cnt` 멤버가 없는 `struct inode_disk`처럼 디스크 데이터 블록과 "비슷한" 대상에도 같은 규칙이 적용됩니다.

즉, 현재 inode 구현이 `struct inode_disk`를 `struct inode` 안에 그대로 포함해 두고 inode를 만들 때 해당 sector를 디스크에서 읽어오는 방식을 사용하고 있으므로, 이 접근 방식은 바꿔야 합니다. inode의 추가 복사본을 유지하면 우리가 여러분의 캐시에 둔 64-block 제한을 우회하게 됩니다.

`struct inode` 안에 inode 데이터를 가리키는 포인터를 저장할 수는 있지만, 그렇게 한다면 운영체제가 동시에 열 수 있는 파일 수가 64개로 제한되지 않도록 매우 조심해야 합니다. inode를 필요할 때 찾는 데 도움이 되는 다른 정보도 저장할 수 있습니다. 마찬가지로, 64개의 각 캐시 엔트리와 함께 몇 가지 메타데이터를 저장할 수도 있습니다.

원한다면 free map의 캐시된 복사본은 메모리에 영구적으로 유지해도 됩니다.
이 복사본은 캐시 크기 제한에 포함되지 않습니다.

`filesys/inode.c`의 `byte_to_sector()`는 해당 sector가 저장소 계층 구조의 어디에 있든 먼저 읽어 오지 않고 `struct inode_disk`를 직접 사용합니다. 이제는 이런 방식이 더 이상 통하지 않습니다. `inode_byte_to_sector()`가 `struct inode_disk`를 사용하기 전에 먼저 캐시에서 가져오도록 바꿔야 합니다.
