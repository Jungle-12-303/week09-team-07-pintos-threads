# 프로젝트 4: 파일 시스템

이전 두 과제에서는 파일 시스템이 내부적으로 어떻게 구현되어 있는지는 크게 신경 쓰지 않은 채 파일 시스템을 광범위하게 사용했습니다. 이번 마지막 과제에서는 파일 시스템의 구현을 개선하게 됩니다. 주로 `filesys` 디렉터리에서 작업하게 됩니다.

프로젝트 4는 프로젝트 2 또는 프로젝트 3 위에 구현할 수 있습니다. 어느 경우든, `filesys` 제출물에서는 프로젝트 2에 필요한 기능이 모두 동작해야 합니다. 프로젝트 3 위에 구현한다면 프로젝트 3의 모든 기능도 동작해야 하며, VM 기능을 활성화하기 위해 `filesys/Make.vars`를 수정해야 합니다. VM을 비활성화하면 점수의 10%를 감점합니다.
page cache (가산점)는 VM 기능이 필요하다는 점에도 유의하세요.

## 배경

### 새로 보는 코드

아래는 여러분에게 아마 처음 보는 파일들입니다. 별도 언급이 없는 한 모두 `filesys` 디렉터리에 있습니다.

- `filesys/fsutil.c`

  > 커널 명령행에서 사용할 수 있는 파일 시스템용 단순 유틸리티입니다.
  >
- `include/filesys/filesys.h`, `filesys/filesys.c`

  > 파일 시스템의 최상위 인터페이스입니다.
  > 소개는 [파일 시스템 사용하기](../03_project2_user_programs/01_introduction.md)를 참고하세요.
  >
- `include/filesys/directory.h`, `filesys/directory.c`

  > 파일 이름을 inode로 변환합니다. 디렉터리 자료구조는 파일로 저장됩니다.
  >
- `include/filesys/inode.h`, `filesys/inode.c`

  > 디스크 위 파일 데이터 배치를 나타내는 자료구조를 관리합니다.
  >
- `include/filesys/fat.h`, `filesys/fat.c`

  > FAT 파일 시스템을 관리합니다.
  >
- `include/filesys/file.h`, `filesys/file.c`

  > 파일 읽기와 쓰기를 디스크 섹터 읽기와 쓰기로 변환합니다.
  >
- `include/filesys/page_cache.h`, `filesys/page_cache.c`

  > VM 기능을 활용하는 page cache 구현입니다. 이 템플릿을 사용하는 것을 강력히 권장합니다. 다만 여러분이 직접 코드를 작성해도 됩니다.
  > VM 플래그를 끄면 이 템플릿을 사용할 수 없다는 점에 유의하세요.
  >
- `include/lib/kernel/bitmap.h`, `lib/kernel/bitmap.c`

  > bitmap 자료구조와, 디스크 파일에 bitmap을 읽고 쓰는 루틴을 제공합니다.
  >

우리 파일 시스템은 Unix와 비슷한 인터페이스를 가지므로 `creat, open, close, read, write, lseek`, `unlink`의 Unix man page도 함께 읽어 보면 좋습니다. 우리 파일 시스템의 호출은 이들과 비슷하지만 완전히 같지는 않습니다. 파일 시스템은 이러한 호출을 디스크 연산으로 변환합니다.

위 코드에는 기본 기능이 모두 들어 있으므로, 이전 두 프로젝트에서 보았듯 파일 시스템은 처음부터 사용할 수 있습니다. 하지만 심각한 한계가 있고, 이번 과제에서는 그 한계를 제거하게 됩니다.

여러분의 작업 대부분은 `filesys`에서 이루어지겠지만, 이전에 구현한 모든 부분과의 상호작용도 염두에 두어야 합니다.

#### 파일 시스템 영속성 테스트

지금까지는 각 테스트가 Pintos를 한 번만 실행했습니다. 하지만 파일 시스템의 중요한 목적 중 하나는 한 번의 부팅에서 다음 부팅까지도 데이터에 계속 접근할 수 있게 하는 것입니다. 따라서 파일 시스템 프로젝트의 테스트는 Pintos를 두 번째로 한 번 더 실행합니다. 두 번째 실행에서는 파일 시스템 안의 모든 파일과 디렉터리를 하나의 파일로 합친 뒤, 그 파일을 Pintos 파일 시스템 밖의 호스트(Unix) 파일 시스템으로 복사합니다.

채점 스크립트는 두 번째 실행에서 복사해 나온 파일의 내용을 기준으로 파일 시스템의 정확성을 검사합니다. 즉, 복사해 낼 파일을 생성하는 Pintos 사용자 프로그램 `tar`를 지원할 만큼 파일 시스템이 충분히 잘 구현되기 전까지는, 확장된 파일 시스템 테스트를 하나도 통과할 수 없습니다. `tar` 프로그램은 요구 사항이 꽤 까다롭습니다(확장 가능한 파일과 하위 디렉터리 지원이 모두 필요합니다). 따라서 어느 정도 작업이 필요합니다. 그 전까지는 추출된 파일 시스템과 관련된 `make check` 오류는 무시해도 됩니다.

덧붙이자면, 이미 짐작했겠지만 파일 시스템 내용을 복사해 내는 데 사용하는 파일 형식은 표준 Unix "tar" 형식입니다. Unix의 tar 프로그램으로 이 파일들을 살펴볼 수 있습니다. 테스트 `t`의 tar 파일 이름은 `t.tar`입니다.
