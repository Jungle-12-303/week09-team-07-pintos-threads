# 04_buffer_cache

### 버퍼 캐시 (가산점)

파일 시스템을 수정해 파일 블록 캐시를 유지하세요. 블록을 읽거나 쓰라는 요청이 들어오면, 먼저 그 블록이 캐시에 있는지 확인하고, 있다면 디스크로 가지 말고 캐시된 데이터를 사용하세요. 없다면 디스크에서 그 블록을 캐시로 가져오고, 필요하다면 더 오래된 엔트리를 축출하세요. 캐시 크기는 64개 섹터를 넘을 수 없습니다.

적어도 "clock" 알고리즘만큼은 되는 캐시 교체 알고리즘을 구현해야 합니다. 일반적으로 데이터보다 메타데이터의 가치가 더 크다는 점을 반영하는 것도 권장합니다. accessed, dirty, 기타 어떤 정보를 조합했을 때 디스크 접근 횟수 기준으로 가장 좋은 성능이 나오는지 실험해 보세요.

원한다면 free map의 캐시된 복사본은 메모리에 영구적으로 유지해도 됩니다. 이 복사본은 캐시 크기 제한에 포함되지 않습니다.

제공된 inode 코드는 `malloc()`으로 할당한 "bounce buffer"를 사용해 디스크의 섹터 단위 인터페이스를 시스템 콜 인터페이스의 바이트 단위 인터페이스로 변환합니다.
이제 이런 bounce buffer는 없애야 합니다. 대신 **버퍼 캐시 안의 섹터로 직접 데이터를 복사해 넣고 꺼내야 합니다.**

캐시는 write-behind 방식이어야 합니다. 즉, dirty block을 즉시 디스크에 쓰지 말고 캐시에 유지해야 합니다. dirty block은 축출될 때 디스크에 기록하세요. write-behind는 크래시가 발생했을 때 파일 시스템을 더 취약하게 만들므로, 이에 더해 dirty한 캐시 블록 전체를 주기적으로 디스크에 다시 기록해야 합니다. 또한 `filesys_done()`에서도 캐시를 디스크에 다시 기록해야 Pintos가 종료될 때 캐시가 flush됩니다.

첫 번째 프로젝트에서 `timer_sleep()`을 제대로 구현해 두었다면, write-behind는 그것을 활용하기에 아주 좋은 예입니다.
그렇지 않다면 덜 일반적인 방법을 구현해도 되지만, busy-waiting이 발생해서는 안 됩니다.

또한 read-ahead도 구현해야 합니다. 즉, 파일의 한 block을 읽을 때 다음 block도 곧 읽힐 가능성이 있다면 자동으로 캐시에 가져와야 합니다. read-ahead는 비동기적으로 수행할 때에만 실질적인 이점이 있습니다. 즉, 어떤 프로세스가 파일의 disk block 1을 요청하면 disk block 1이 읽힐 때까지는 그 프로세스가 block되어야 하지만, 그 읽기가 끝나면 제어는 즉시 프로세스로 돌아가야 합니다. disk block 2에 대한 read-ahead 요청은 백그라운드에서 비동기적으로 처리되어야 합니다.
이를 위해 I/O 전용 스레드를 만들어야 할 수도 있습니다. 디스크 I/O가 필요할 때 해당 스레드가 전용 스레드에게 요청을 넘기는 방식입니다.

#### 구현 힌트

Linux가 하듯이 이 기능을 가상 메모리 서브시스템을 통해 구현할 수도 있습니다. 새로운 vm type(예: "pagecache")를 도입하고, `swap_in`, `swap_out`, `destroy` 인터페이스를 통해 버퍼 캐시를 관리할 수 있습니다.
이 설계를 선택하면 `SECTOR_SIZE` 단위가 아니라 `PAGE_SIZE` 단위로 캐시를 관리할 수 있습니다.
file backed page의 동기화 문제는 `mmaped` 영역을 그 페이지들의 캐시로 사용하는 방식으로 해결할 수 있습니다.

**캐시는 설계 초기에 통합하는 것을 권장합니다.**  과거에는 많은 팀이 설계 후반부에 캐시를 덧붙이려고 했습니다.
이것은 매우 어렵습니다. 그런 팀들은 종종 대부분의 테스트, 심지어 거의 모든 테스트를 통과하지 못한 프로젝트를 제출하곤 했습니다.
