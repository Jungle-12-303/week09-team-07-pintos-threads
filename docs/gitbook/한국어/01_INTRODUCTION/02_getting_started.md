# 02_getting_started

# 시작하기

시작하려면 먼저 Pintos를 빌드할 수 있는 머신에 로그인해야 합니다.
교수진이 제공한 Linux 머신이나 여러분이 보유한 Solaris 또는 Linux 머신을 사용할 수 있습니다. 하지만 우리는 제공된 머신에서 여러분의 코드를 테스트하며, 여기의 지침도 이 환경을 전제로 작성되어 있습니다. 자체 머신에 Pintos를 설치하고 작업하는 방법에 대해서는 지원을 제공할 수 없습니다. 아래와 같은 서버 환경이 각 학생에게 제공됩니다.

이제 다음을 실행해 소스 코드를 `pintos`라는 디렉터리로 가져올 수 있습니다:

```/bin/sh
git clone https://github.com/casys-kaist/pintos-kaist
```

하지만 소스 코드는 **프로젝트 기간이 시작되기 전까지** 변경될 가능성이 있으므로, [저장소를 복제해 두는 것](https://help.github.com/en/github/creating-cloning-and-archiving-repositories/duplicating-a-repository)을 권장합니다.
템플릿 코드가 변경되면 [쉽게 동기화할 수 있습니다](https://help.github.com/en/github/collaborating-with-issues-and-pull-requests/syncing-a-fork).
**공개 포크는 만들지 마세요.** 
템플릿 코드에 치명적인 버그가 있는 경우를 제외하면, 프로젝트 기간 중에는 소스 코드를 변경하지 않습니다.

참고로, 저희는 `Ubuntu 16.04.6 LTS`의 gcc (`gcc (Ubuntu 7.4.0-1ubuntu1~16.04~ppa1) 7.4.0`)와 qemu-system-x86_64
(`QEMU emulator version 2.5.0 (Debian 1:2.5+dfsg-5ubuntu10.43)`) 환경에서 pintos 코드와 정답 코드를 테스트했습니다. 이는 제공되는 Linux 머신과 동일합니다.

소스 코드를 받아온 뒤에는, pintos 루트 디렉터리에서 아래 명령으로 환경을 설정할 수 있습니다:

```sh
$ source ./activate
```

이 줄을 홈 디렉터리의 `.bashrc`에 추가해 두는 것이 좋습니다.
그렇지 않으면 로그인할 때마다 매번 직접 입력해야 합니다.

## 소스 트리 개요

안에 무엇이 들어 있는지 살펴보겠습니다. `pintos/` 아래에는 다음과 같은 디렉터리 구조가 있어야 합니다.

- `threads/`: 기본 커널의 소스 코드입니다. 프로젝트 1부터 이 코드를 수정하게 됩니다.
- `userprog/`: 사용자 프로그램 로더의 소스 코드입니다. 프로젝트 2부터 이 코드를 수정하게 됩니다.
- `vm/`: 거의 비어 있는 디렉터리입니다. 프로젝트 3에서 여기서 가상 메모리를 구현합니다.
- `filesys/`: 기본 파일 시스템의 소스 코드입니다. 프로젝트 2부터 이 파일 시스템을 사용하지만, 프로젝트 4 전까지는 수정하지 않습니다.
- `devices/`: 키보드, 타이머, 디스크 등 I/O 장치 인터페이스용 소스 코드입니다. 프로젝트 1에서 타이머 구현을 수정하게 됩니다. 그 외에는 이 코드를 바꿀 일이 거의 없습니다.
- `lib/`: 표준 C 라이브러리 일부를 구현한 코드입니다. 이 디렉터리의 코드는 Pintos 커널과, 프로젝트 2부터는 그 위에서 실행되는 사용자 프로그램에도 함께 컴파일됩니다. 이 코드를 수정할 일은 거의 없습니다.
- `include/lib/kernel/`: Pintos 커널에만 포함되는 C 라이브러리 부분입니다. 커널 코드에서 자유롭게 사용할 수 있는 몇 가지 자료구조 구현도 여기에 포함되어 있습니다. 예를 들면 bitmap, 이중 연결 리스트, 해시 테이블이 있습니다. 커널에서는 이 디렉터리의 헤더를 `#include <...>` 표기법으로 포함할 수 있습니다.
- `include/lib/user/`: Pintos 사용자 프로그램에만 포함되는 C 라이브러리 부분입니다. 사용자 프로그램에서는 이 디렉터리의 헤더를 `#include <...>` 표기법으로 포함할 수 있습니다.
- `tests/`: 각 프로젝트용 테스트입니다. 제출물을 테스트하는 데 도움이 된다면 이 코드를 수정해도 되지만, 실제 테스트를 실행하기 전에 우리는 이 디렉터리를 원본으로 교체합니다.
- `examples/`: 프로젝트 2부터 사용할 예제 사용자 프로그램입니다.
- `include/`: 헤더 파일(`*.h`)의 소스 코드입니다.

## Pintos 빌드하기

다음 단계로, 첫 번째 프로젝트를 위해 제공된 소스 코드를 빌드해 보겠습니다. 먼저 `threads` 디렉터리로 `cd`한 뒤 `make` 명령을 실행하세요.
그러면 `threads` 아래에 `build` 디렉터리가 만들어지고, 그 안에 `Makefile`과 몇 개의 하위 디렉터리가 채워진 뒤, 그 내부에서 커널이 빌드됩니다.

빌드가 끝나면 `build` 디렉터리에서 특히 눈여겨볼 파일은 다음과 같습니다.

- `Makefile`: `pintos/src/Makefile.build`를 복사한 파일입니다. 커널을 어떻게 빌드하는지 설명합니다.
- `kernel.o`: 전체 커널의 오브젝트 파일입니다. 각 개별 커널 소스 파일에서 컴파일된 오브젝트 파일들을 하나의 오브젝트 파일로 링크한 결과입니다. 디버그 정보가 들어 있으므로 GDB
  ([GDB](../06_appendix/06_debugging_tools.md#gdb))나 backtrace
  ([백트레이스](../06_appendix/06_debugging_tools.md#backtraces))를 여기에 대해 실행할 수 있습니다.
- `kernel.bin`: 커널의 메모리 이미지, 즉 Pintos 커널을 실행하기 위해 메모리에 적재되는 정확한 바이트열입니다. 이것은 `kernel.o`에서 디버그 정보를 제거한 버전입니다. 이렇게 하면 공간을 크게 절약할 수 있고, 그 결과 커널 로더 설계상 부과된 512 kB 크기 제한에 커널이 걸리는 일을 막을 수 있습니다.
- `loader.bin`: 커널 로더용 메모리 이미지입니다. 어셈블리 언어로 작성된 작은 코드 조각으로, 디스크에서 커널을 메모리로 읽어들인 뒤 실행을 시작합니다. 크기는 정확히 512바이트이며, 이는 PC BIOS가 정한 고정 크기입니다.
  `build`의 하위 디렉터리에는 컴파일러가 생성한 오브젝트 파일 (`.o`)과 의존성 파일 (`.d`)도 들어 있습니다. 의존성 파일은 다른 소스 파일이나 헤더 파일이 바뀌었을 때 어떤 소스 파일을 다시 컴파일해야 하는지 make에 알려 줍니다.

## Pintos 실행하기

시뮬레이터에서 Pintos를 편리하게 실행할 수 있도록 `pintos`라는 프로그램을 제공합니다. 가장 단순한 경우에는 `pintos argument....` 형태로 실행할 수 있습니다.
각 인자는 Pintos 커널에 전달되어 해당 동작을 수행하게 됩니다.
직접 실행해 보세요. 먼저 새로 생성된 `build` 디렉터리로 `cd`한 뒤 `pintos run alarm-multiple` 명령을 실행합니다. 그러면 `run alarm-multiple` 인자가 Pintos 커널에 전달됩니다. 여기서 `run`은 커널에게 테스트를 실행하라고 지시하는 명령이고, `alarm-multiple`은 실행할 테스트 이름입니다. Pintos는 부팅한 뒤 `alarm-multiple` 테스트 프로그램을 실행하며, 몇 화면 분량의 텍스트를 출력합니다.

명령행에서 리다이렉션해 시리얼 출력을 파일로 남길 수도 있습니다. 예를 들어 `pintos -- run alarm-multiple > logfile`와 같이 사용하면 됩니다. `pintos` 프로그램은 QEMU나 가상 하드웨어를 설정하기 위한 여러 옵션을 제공합니다. 옵션을 지정하는 경우, Pintos 커널에 전달할 명령보다 앞에 두어야 하며 `--`로 구분해야 합니다. 즉 전체 명령은 `pintos option... -- argument....` 형태가 됩니다. 사용 가능한 옵션 목록은 인자 없이 `pintos`를 실행해 확인할 수 있습니다. 옵션에는 VM 출력을 어떻게 표시할지 고르는 것도 포함됩니다. `-v`를 사용하면 VGA 디스플레이를 끄고, `-s`를 사용하면 stdin에서의 시리얼 입력과 stdout으로의 출력을 억제합니다. Pintos 커널에는 `run` 외에도 다른 명령과 옵션이 있습니다. 지금은 크게 중요하지 않지만, `pintos -h`처럼 `-h`를 사용하면 목록을 볼 수 있습니다.
