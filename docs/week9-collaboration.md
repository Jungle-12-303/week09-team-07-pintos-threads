# 9주차 Pintos Threads 협업 문서

## 1. 문서 목적

이 문서는 9주차 Pintos `Project 1: Threads`를 3인 팀이 함께 진행할 때의 협업 기준을 정리한 공식 문서다. `local/` 폴더의 자료는 개인 참고용으로 두고, 팀이 실제로 합의해 따를 내용은 `docs/` 아래 문서로 관리한다.

이번 주의 목표는 "가능한 많은 코드를 빠르게 합치는 것"이 아니라, 스케줄링과 동기화의 핵심 개념을 팀원 각자가 설명할 수 있을 정도로 이해하고 자기 것으로 만드는 것이다. 테스트 통과는 중요하지만, 테스트 통과만으로 병합하지 않는다.

구체적인 구현 순서와 병합 기준은 [9주차 구현 계획 문서](week9-implementation-plan.md)에서 별도로 관리한다.

## 2. 9주차 협업 원칙

- 팀 최종 결과물은 하나의 공용 코드베이스로 관리한다.
- 구현은 테스트를 맞추기 위한 땜질보다 설계 의도와 동작 원리를 설명할 수 있는 방향으로 진행한다.
- 코드를 작성한 사람은 자신이 수정한 함수, 자료구조, 인터럽트/동기화 가정을 설명할 수 있어야 한다.
- 테스트가 통과해도 "왜 통과하는지" 설명하지 못하면 병합하지 않는다.
- AI 도구는 문서 요약, 개념 정리, 디버깅 가설 정리에 활용할 수 있지만, 결과를 그대로 복사해 병합하지 않는다.
- AI의 도움을 받았더라도 최종 코드와 설명 책임은 작성자에게 있다.

## 3. 브랜치 전략

| 브랜치 | 역할 | 운영 원칙 |
|------|------|------|
| `main` | 최종 제출 기준 브랜치 | 직접 push 금지, 충분히 검증된 상태만 반영 |
| `dev` | 팀 통합 브랜치 | PR 리뷰와 관련 테스트 확인 후 병합 |
| 개인 브랜치 | 개인 작업 브랜치 | `changwon`, `hyungmin`, `donghyun`처럼 팀원 단위로 사용 |

기본 흐름은 다음과 같다.

1. GitHub Projects에서 작업 이슈를 확인하고 담당 범위를 정한다.
2. 각자 개인 브랜치에서 작업한다.
3. 관련 테스트, 수정 파일, 설계 의도를 정리한 뒤 `dev`로 PR을 올린다.
4. 팀원이 로직과 테스트 결과를 검토한 뒤 `dev`에 병합한다.
5. `dev`가 충분히 안정적일 때만 `main`에 반영한다.

원칙적으로 `main`에는 직접 작업하지 않는다. 긴급 수정이 필요한 경우에도 팀 합의 후 `dev`를 거쳐 반영한다.

## 4. 이슈 보드 운영 방식

9주차 이슈는 GitHub Projects 보드를 기준으로 관리한다.

Project URL: https://github.com/orgs/Jungle-12-303/projects/4

- 공통 이슈: 목표 설정, AI 활용 원칙, WIL, 협업 규칙 정리
- 학습 이슈: KAIST GitBook, Project 1 문서, FAQ 및 동기화 문서 학습
- 테스트 이슈: `alarm`, `priority`, `mlfqs` 테스트 단위 작업

이슈를 나눌 때는 "사람 수대로 균등 분배"보다 "관련 함수와 실패 원인이 비슷한 테스트끼리 묶기"를 우선한다. 즉, 앞으로의 업무 계획도 테스트 이름 기준으로만 쪼개지 않고, 실제 코드 연관성과 설계 결합도를 함께 고려해 분할한다.

## 5. 초기 업무 분장

초기 `alarm` 작업은 테스트 기반으로 아래처럼 나눈다.

| 담당자 | 초기 담당 테스트 | 중점 확인 포인트 |
|------|------|------|
| `changwon` | `alarm-single`, `alarm-multiple` | 기본 sleep/wakeup 흐름, wakeup tick 계산 |
| `hyungmin` | `alarm-simultaneous`, `alarm-priority` | 같은 tick에서의 다중 wakeup, wakeup 이후 priority 반영 |
| `donghyun` | `alarm-zero`, `alarm-negative` | 경계값 처리, `timer_sleep()`의 API 동작 |

단, `alarm` 구현은 결국 같은 설계를 공유한다. `sleeping list`, `wakeup_tick`, `timer_interrupt()`, `thread_unblock()` 흐름은 서로 강하게 연결되어 있으므로, 개인 담당 테스트가 달라도 병합 전에는 3명이 함께 전체 흐름을 리뷰한다.

## 6. 이후 업무 분할 원칙

이후 작업은 관련 테스트 묶음과 수정 파일 결합도를 기준으로 분할한다. `alarm`, `priority`, `donation`, `mlfqs`처럼 설계 단위가 갈리는 영역은 묶어서 보고, `pintos/threads/thread.c`, `pintos/threads/synch.c`, `pintos/devices/timer.c`처럼 충돌 가능성이 큰 파일은 병렬 수정 전에 공유한다.

상세 작업 묶음 표와 병합 전 확인 포인트는 [9주차 구현 계획 문서](week9-implementation-plan.md)의 `## 6. 이후 작업 분할 기준`을 따른다.

## 7. PR, 리뷰, 병합 기준

모든 PR에는 최소한 아래 내용이 들어가야 한다.

- 관련 이슈 또는 테스트 이름
- 수정한 파일과 핵심 함수
- 통과한 테스트와 아직 남은 테스트
- 구현 의도 또는 설계 가정
- 남아 있는 리스크 또는 확인이 필요한 부분

`dev` 병합 기준은 다음과 같다.

- 관련 테스트가 재현 가능하게 통과해야 한다.
- 작성자가 수정 이유와 동작 원리를 설명할 수 있어야 한다.
- 팀원 1명 이상이 리뷰해야 한다.
- `thread.c`, `synch.c`, `timer.c`, 헤더 구조 변경은 가능하면 함께 보고 병합한다.
- race condition, priority inversion, donation 회수, MLFQS 계산식처럼 핵심 개념이 걸린 변경은 구두 설명 또는 코어타임 리뷰를 거친다.

다음 경우에는 병합하지 않는다.

- 테스트는 통과했지만 이유를 설명하지 못하는 경우
- 관련 테스트 외 회귀 영향이 의심되는데 확인이 없는 경우
- AI 제안이나 외부 코드를 그대로 가져와 이해 없이 붙인 경우

## 8. 커밋 메시지와 PR 작성 규칙

- 커밋은 한 가지 의도만 담고, 기능명이나 테스트명을 포함해 작성한다.
- 커밋 메시지는 "무엇을 왜 바꿨는지"가 보이게 작성한다.
- `fix`, `update`처럼 의미가 약한 단어만 단독으로 쓰지 않는다.
- 커밋 예시: `alarm: fix simultaneous wakeup ordering`
- 커밋 예시: `priority: yield after lowering current thread priority`
- PR 본문에는 관련 테스트, 수정 파일, 핵심 함수, 구현 의도, 통과/미통과 테스트, 남은 리스크를 함께 적는다.
- PR 설명은 결과만 적지 말고, 어떤 설계 변경이 어떤 테스트에 영향을 줬는지 연결해서 쓴다.

## 9. 테스트 운영 원칙

테스트는 "끝났는지 체크하는 용도"가 아니라 "설계 가설이 맞는지 확인하는 도구"로 사용한다.

- 실패 테스트를 공유할 때는 테스트 이름, 재현 명령, 기대 결과, 실제 결과, 최근 수정 내용을 함께 남긴다.
- 한 번에 여러 테스트가 깨지면 의존성이 앞선 단계부터 본다.
- `priority-donate-*`가 실패할 때 `priority-preempt`, `priority-fifo`, `priority-sema`가 먼저 안정적인지 확인한다.
- `mlfqs-*`는 `priority`와 `donation` 흐름과 분리해서 점검한다.
- 관련 없는 대규모 수정으로 여러 테스트를 동시에 건드리지 않는다.

## 10. 소통과 학습 공유

- 막힌 테스트는 오래 끌지 말고 바로 공유한다.
- 코어타임이나 스크럼에서는 "무엇을 바꿨는지"보다 "왜 그렇게 설계했는지"를 우선 설명한다.
- 이해가 부족한 상태의 병합보다, 늦더라도 팀 전체가 설계를 따라갈 수 있는 병합을 우선한다.
- WIL에는 단순 결과보다 배운 개념, 실패 원인, 수정 근거를 남긴다.

## 11. 발표와 기록 원칙

- 주간 공유 발표 자료는 팀 기준 하나의 문서로 통합한다.
- 구현 내용, 실패 테스트, 트러블슈팅, 설계 변경 이유를 주중에 계속 누적한다.
- 마지막 날에 발표 자료를 새로 만드는 대신, PR과 WIL에서 쌓인 근거를 재사용한다.
- 개인 발표가 있더라도 팀 문서 기준의 설명 흐름은 맞춘다.

## 12. 9주차 완료 기준

9주차 작업이 잘 진행되고 있다고 볼 수 있는 기준은 다음과 같다.

- 각 담당자는 자신이 맡은 테스트와 관련 함수의 동작을 설명할 수 있다.
- 팀원은 `alarm`, `priority`, `donation`, `mlfqs`를 구현 순서와 의존성 기준으로 설명할 수 있다.
- `dev`에 병합되는 변경은 관련 테스트와 설계 설명이 함께 남는다.
- 코드, 테스트, 문서, 발표 자료가 따로 놀지 않고 같은 근거를 공유한다.
